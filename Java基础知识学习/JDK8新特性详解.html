一、Lambda 表达式 —— 行为参数化

Lambda 的本质是一段可传递的行为（代码片段），让你能像传数据一样传"做什么"。

1.1 从匿名内部类到 Lambda

JDK 7：匿名内部类

new Thread(new Runnable() {
    public void run() {
        System.out.println("hello");
    }
}).start();

6 行，大量模板代码

JDK 8：Lambda

new Thread(() ->
    System.out.println("hello")
).start();

3 行，只关注行为本身

1.2 Lambda 语法

// 基本格式：(参数) -> { 方法体 }

// 无参，无返回值
() -> System.out.println("hello")

// 一个参数（可省略括号），有返回值
x -> x * 2

// 多个参数，多行方法体
(x, y) -> {
    int result = x + y;
    return result;
}

// 类型声明（通常省略，靠类型推断）
(String s) -> s.length()
Lambda 的本质：它是函数式接口的实例。一个 Lambda 表达式能赋值给一个函数式接口类型的变量，编译器自动推断目标类型（类型推断）。

1.3 Lambda 中使用局部变量 —— 必须"有效 final"

int num = 10;                // 有效 final（未被修改过）
Runnable r = () -> System.out.println(num);
// num = 20; ← 如果加了这行，上面的 Lambda 就编译失败

原因：Lambda 本质上是匿名内部类的语法糖，局部变量被拷贝到 Lambda 表达式中。如果允许修改，会导致数据不一致。

二、函数式接口 —— 四大核心接口

函数式接口：只有一个抽象方法的接口（可以有默认方法和静态方法）。用 @FunctionalInterface 注解标记。

JDK 8 内置的四大核心函数式接口： ┌─────────────────────────────────────────────────────┐ │ Supplier<T> 供给型 () -> T │ │ Consumer<T> 消费型 T -> void │ │ Function<T,R> 函数型 T -> R │ │ Predicate<T> 断言型 T -> boolean │ └─────────────────────────────────────────────────────┘ 
// Supplier —— 无中生有（工厂、懒加载）
Supplier<String> supplier = () -> "hello";
String s = supplier.get();  // "hello"
// 典型场景：Optional.orElseGet(Supplier)

// Consumer —— 消费掉（forEach、回调）
Consumer<String> consumer = str -> System.out.println(str);
consumer.accept("hello");
// 典型场景：list.forEach(consumer)、peek(consumer)

// Function —— 转换（map）
Function<String, Integer> func = String::length;
Integer len = func.apply("hello");  // 5
// 典型场景：stream.map(function)

// Predicate —— 判断（filter）
Predicate<String> pred = str -> str.length() > 3;
boolean b = pred.test("hi");  // false
// 典型场景：stream.filter(predicate)

// 组合使用：and / or / negate / andThen / compose
Predicate<String> notEmpty = str -> !str.isEmpty();
Predicate<String> longEnough = str -> str.length() > 5;
Predicate<String> valid = notEmpty.and(longEnough);  // 同时满足两个条件

2.1 其他常用函数式接口

接口签名用途
BiFunction<T,U,R>(T, U) → R两个参数、有返回值
BiConsumer<T,U>(T, U) → void两个参数的消费者
UnaryOperator<T>T → T一元运算（入参出参同类型），extends Function
BinaryOperator<T>(T, T) → T二元运算（reduce 使用）
IntFunction<R>int → R避免装箱（Int/Long/Double 变种）

三、方法引用与构造器引用

当 Lambda 表达式的内容只是调用一个已有方法时，可以简写为方法引用。方法引用比 Lambda 更简洁，可读性更强。

// 四种方法引用形式：

// ① 静态方法引用  ClassName::staticMethod
Function<String, Integer> f1 = Integer::parseInt;  // s -> Integer.parseInt(s)

// ② 实例方法引用  instance::method
String prefix = "Hello ";
Function<String, String> f2 = prefix::concat;    // s -> prefix.concat(s)

// ③ 对象方法引用  ClassName::instanceMethod
Function<String, Integer> f3 = String::length;    // s -> s.length()
//    ↑ 注意：第一个参数成为方法调用的"接收者"

// ④ 构造器引用  ClassName::new
Supplier<ArrayList<String>> f4 = ArrayList::new;    // () -> new ArrayList<>()
Function<Integer, ArrayList<String>> f5 = ArrayList::new; // size -> new ArrayList<>(size)
什么时候用方法引用代替 Lambda？
当 Lambda 体中只有一个方法调用且参数完全匹配时。例如 x -> String.valueOf(x) → String::valueOf。更可读，IDE 也会提示转换。

四、Stream API —— 流式编程

Stream 是 JDK 8 最重要的特性，它提供了一套声明式处理集合数据的工具。核心思想：做什么，而不是怎么做。

4.1 Stream 的三大组成部分

Stream 操作流水线： 数据源 → 中间操作(可多个) → 终端操作(触发计算) ┌──────────┐ ┌──────────────┐ ┌──────────────┐ │ ① 创建流 │ ──→ │ ② 中间操作(N个) │ ──→ │ ③ 终端操作 │ │ list │ │ filter │ │ collect │ │ Stream.of │ │ map │ │ forEach │ │ Arrays. │ │ sorted │ │ reduce │ │ stream │ │ distinct │ │ count │ │ │ │ limit/skip │ │ findFirst │ └──────────┘ └──────────────┘ └──────────────┘ (懒) (懒, 可链式) (触发计算)
关键特性：
· 中间操作惰性求值：不调用终端操作，中间操作不执行
· Stream 不存储数据：只是数据的"视图"，不改变原集合
· Stream 只能消费一次：终端操作执行后，流就关闭了

4.2 创建 Stream 的方式

// ① 从集合
List<String> list = Arrays.asList("a", "b", "c");
Stream<String> s1 = list.stream();           // 串行流
Stream<String> s2 = list.parallelStream();  // 并行流

// ② 从数组
int[] arr = {1, 2, 3};
IntStream s3 = Arrays.stream(arr);

// ③ 直接创建
Stream<String> s4 = Stream.of("a", "b", "c");
Stream<Integer> s5 = Stream.iterate(0, n -> n + 2); // 无限流 → 必须 limit
Stream<Double> s6  = Stream.generate(Math::random);  // 无限流

// ④ 文件行流
Stream<String> lines = Files.lines(Paths.get("file.txt"));

4.3 常用中间操作

// ===== filter：过滤 =====
list.stream().filter(s -> s.length() > 3)

// ===== map：转换 =====
list.stream().map(String::toUpperCase)           // 每个元素映射为新值
list.stream().flatMap(s -> Arrays.stream(s.split(",")))  // 扁平化：[[a,b], [c,d]] → [a,b,c,d]

// ===== sorted：排序 =====
list.stream().sorted()                             // 自然排序
list.stream().sorted(Comparator.comparing(User::getAge).reversed())

// ===== distinct / limit / skip =====
list.stream().distinct()                         // 去重（基于 equals）
list.stream().limit(10)                           // 取前 10 个
list.stream().skip(5)                            // 跳过前 5 个

// ===== peek：调试用（看一眼但不消费）=====
list.stream().peek(System.out::println)           // 每个元素都打印

4.4 map vs flatMap —— 最容易混淆的两个方法

// 场景：把 ["hello", "world"] 拆成 ['h','e','l','l','o','w','o','r','l','d']

// ❌ 错误：用 map → 得到的是 Stream<Stream<Character>>
list.stream().map(s -> s.chars().mapToObj(c -> (char) c))
//  → Stream<Stream<Character>> 不是我们想要的

// ✅ 正确：用 flatMap → 扁平化
list.stream().flatMap(s -> s.chars().mapToObj(c -> (char) c))
//  → Stream<Character>

// 口诀：map = 一一映射，flatMap = 一对多映射 + 压扁

4.5 常用终端操作

// ===== 收集：collect =====
List<String> newList = list.stream().collect(Collectors.toList());

// ===== 遍历：forEach / forEachOrdered =====
list.stream().forEach(System.out::println);     // 并行时不保证顺序
list.stream().forEachOrdered(System.out::println); // 保证顺序

// ===== 归约：reduce =====
int sum = IntStream.of(1, 2, 3).reduce(0, Integer::sum);  // 6
// reduce(初始值, 累加器)

// ===== 查找：findFirst / findAny =====
Optional<String> first = list.stream().findFirst();  // 第一个
Optional<String> any   = list.stream().findAny();    // 任意一个（并行时更快）

// ===== 匹配：allMatch / anyMatch / noneMatch =====
boolean all = list.stream().allMatch(s -> s.length() > 2);
boolean any = list.stream().anyMatch(s -> s.startsWith("A"));

// ===== 聚合：count / max / min =====
long count = list.stream().count();
Optional<Integer> max = list.stream().map(String::length).max(Integer::compareTo);

4.6 数值流 —— 避免装箱开销

// IntStream / LongStream / DoubleStream 专门处理基本类型，避免装箱

IntStream.range(1, 100)                  // [1, 100)
IntStream.rangeClosed(1, 100)            // [1, 100]

// 普通 Stream → 数值流
list.stream().mapToInt(String::length)   // Stream<String> → IntStream

// 数值流 → 普通 Stream（装箱）
IntStream.of(1, 2, 3).boxed()       // IntStream → Stream<Integer>

// 数值流特有方法：sum / average / summaryStatistics
int sum = IntStream.of(1, 2, 3).sum();
IntSummaryStatistics stats = IntStream.of(1, 2, 3).summaryStatistics();
// stats.getCount(), getSum(), getMin(), getMax(), getAverage()

4.7 并行流 —— 一行代码实现并行

// 串行 → 并行只需一个方法调用
list.stream().filter(...).collect(...);                     // 串行
list.parallelStream().filter(...).collect(...);            // 并行
list.stream().parallel().filter(...).sequential()...; // 动态切换
并行流慎用场景：
· 数据量小（< 1000 条）→ 线程调度开销 > 并行收益
· 有状态中间操作（distinct、sorted）→ 并行效率低
· 操作有副作用 → 线程不安全
· 底层用 ForkJoinPool.commonPool()，注意别阻塞

五、Collectors —— 收集器大全

Collectors 是 Stream 的"终点站工具箱"，提供了几十种预定义的收集方式。

// ===== 转集合 =====
.collect(Collectors.toList())                      // → ArrayList
.collect(Collectors.toSet())                       // → HashSet
.collect(Collectors.toCollection(TreeSet::new))   // → 指定集合类型

// ===== 转 Map =====
.collect(Collectors.toMap(
    User::getId,           // keyMapper
    Function.identity(),   // valueMapper
    (old, new) -> new      // mergeFunction: key 冲突时选新值
))
// JDK 21: .collect(Collectors.toMap(User::id, Function.identity())) 无需 merge 函数

// ===== 分组 =====
.collect(Collectors.groupingBy(User::getDept))               // → Map<String, List<User>>
.collect(Collectors.groupingBy(User::getDept,
    Collectors.counting()))                                      // → Map<String, Long> 统计人数
.collect(Collectors.groupingBy(User::getDept,
    Collectors.mapping(User::getName, Collectors.toList())))    // 分组后提取某个字段

// ===== 分区（只能是 true / false 两组）=====
.collect(Collectors.partitioningBy(user -> user.getAge() >= 18))
// → Map<Boolean, List<User>>

// ===== 拼接 =====
.collect(Collectors.joining())                    // → "abc"
.collect(Collectors.joining(", "))                // → "a, b, c"
.collect(Collectors.joining(", ", "[", "]"))    // → "[a, b, c]"

// ===== 统计 =====
.collect(Collectors.counting())
.collect(Collectors.summingInt(User::getAge))
.collect(Collectors.averagingInt(User::getAge))
.collect(Collectors.summarizingInt(User::getAge))  // count + sum + min + avg + max

// ===== 最大 / 最小 =====
.collect(Collectors.maxBy(Comparator.comparing(User::getAge)))
.collect(Collectors.minBy(Comparator.comparing(User::getAge)))

// ===== reducing (一般直接用 reduce 或特定方法，reducing 较底层) =====
.collect(Collectors.reducing(0, User::getAge, Integer::sum))

5.1 实战：嵌套分组 + 统计

// 场景：按部门分组，再按城市分组，统计平均工资
Map<String, Map<String, Double>> result = users.stream()
    .collect(Collectors.groupingBy(
        User::getDept,                                    // 第一层：部门
        Collectors.groupingBy(
            User::getCity,                                // 第二层：城市
            Collectors.averagingDouble(User::getSalary)    // 第三层：平均工资
        )
    ));

六、Optional —— 优雅处理 null

Optional 是一个容器对象，可能包含也可能不包含非 null 值。它的设计目标是取代 null 检查，减少 NPE。

6.1 创建 Optional

// 三种创建方式
Optional<String> opt1 = Optional.of("hello");         // 非 null 值，null 则抛 NPE
Optional<String> opt2 = Optional.ofNullable(getValue()); // 可以为 null（最常用）
Optional<String> opt3 = Optional.empty();               // 空 Optional

6.2 提取值

方法有值时无值时推荐？
get()返回值抛 NoSuchElementException❌ 别用
orElse(T other)返回值返回 other⚠️ 即使有值也会执行 other 的构造
orElseGet(Supplier)返回值执行 Supplier 返回✅ 推荐，懒加载
orElseThrow(Supplier)返回值抛出自定义异常✅ 业务异常场景
ifPresent(Consumer)执行 Consumer什么都不做✅
ifPresentOrElse(Consumer, Runnable)执行 Consumer执行 Runnable✅ JDK 9+

6.3 orElse vs orElseGet —— 一个经典坑

// ❌ 错误用法：即使 value 不为 null，expensiveOperation() 也会被执行！
String result = optional.orElse(expensiveOperation());

// ✅ 正确用法：只有 value 为 null 时才会执行
String result = optional.orElseGet(() -> expensiveOperation());

// 原因：Java 方法调用会先计算参数的值，再传入方法。
//       orElse(x) 中的 x 在调用前就已经计算好了。

6.4 链式操作

// ===== map：转换值 =====
Optional<String> name = userOpt.map(User::getName);

// ===== flatMap：防止 Optional 嵌套 =====
// user.getAddress() 返回 Optional<Address>
// 如果直接 map → Optional<Optional<Address>> 嵌套！
Optional<String> city = userOpt
    .flatMap(User::getAddress)      // 扁平化，得到 Optional<Address>
    .map(Address::getCity);         // 继续转换

// ===== filter：过滤 =====
Optional<User> adult = userOpt.filter(u -> u.getAge() >= 18);

// ===== 实战：优雅处理级联调用 =====
// JDK 7：
String city = null;
if (user != null) {
    Address addr = user.getAddress();
    if (addr != null) {
        city = addr.getCity();
    }
}

// JDK 8：
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
Optional 使用原则（来自 Optional 作者 Brian Goetz）：
· 不要用 Optional 做字段类型（不是 Serializable）
· 不要用 Optional 做方法参数
· Optional 只应该作为返回值使用
· 不要调用 get()，它只是给旧代码的"逃生门"

七、接口默认方法与静态方法

JDK 8 之前，接口只能有抽象方法。JDK 8 允许接口有默认方法（default method）和静态方法（static method）。

public interface Vehicle {

    // 抽象方法（必须实现）
    void start();

    // 默认方法（子类可选覆盖）
    default void honk() {
        System.out.println("嘀嘀嘀");
    }

    // 静态方法（只能通过接口名调用）
    static boolean isFast(int speed) {
        return speed > 120;
    }
}

// 多继承冲突：实现两个接口，有同名默认方法 → 必须手动选择
public class Car implements A, B {
    public void doSomething() {
        A.super.doSomething();  // 明确选 A 的方法
    }
}

// 类优先：类中的方法优先级 > 接口默认方法

为什么引入默认方法？主要是为了给 Collection 接口加 stream() 方法。如果直接加抽象方法，所有 Collection 的实现类（包括你写的）全都编译报错。default 方法可以不破坏已有实现。

接口 vs 抽象类（JDK 8 之后）：
接口仍然不能有构造器、不能有实例字段、不能有非 public 方法。区别的核心在于"能不能有状态"，接口只能有行为（方法），不能有状态（字段）。

八、新的日期时间 API —— java.time

JDK 8 之前的 java.util.Date 和 Calendar 设计糟糕（可变、非线程安全、月份从 0 开始、Date 同时表示日期+时间...）。JDK 8 在 Joda-Time 的基础上设计了全新的 java.time 包。

8.1 核心类速览

类描述示例
LocalDate日期（年月日）2024-03-15
LocalTime时间（时分秒）14:30:00
LocalDateTime日期+时间2024-03-15T14:30:00
ZonedDateTime带时区的日期时间2024-03-15T14:30:00+08:00[Asia/Shanghai]
Instant时间戳（UTC 时间线上的一个点）2024-03-15T06:30:00Z
Duration时间间隔（时分秒纳秒）PT2H30M（2小时30分钟）
Period日期间隔（年月日）P1Y2M3D（1年2个月3天）
DateTimeFormatter格式化器（线程安全！）yyyy-MM-dd HH:mm:ss

8.2 常用操作

// ===== 创建 =====
LocalDate today = LocalDate.now();
LocalDate date  = LocalDate.of(2024, 3, 15);    // 不是 new ! 月份是 3 不是 2 !
LocalDateTime dt = LocalDateTime.of(2024, 3, 15, 14, 30);

// ===== 修改（所有方法返回新对象，原对象不变 —— 不可变，线程安全）=====
LocalDate tomorrow = today.plusDays(1);
LocalDate lastMonth = today.minusMonths(1);
LocalDate firstDay  = today.withDayOfMonth(1);
LocalDate adjusted  = today.with(TemporalAdjusters.lastDayOfMonth());

// ===== 计算差距 =====
long days = ChronoUnit.DAYS.between(date1, date2);
Period period = Period.between(date1, date2);
// period.getYears(), getMonths(), getDays()

Duration duration = Duration.between(time1, time2);
// duration.toHours(), toMinutes(), getSeconds()

// ===== 时间戳（Instant）=====
Instant now = Instant.now();
long millis = now.toEpochMilli();     // 毫秒时间戳
long seconds = now.getEpochSecond();    // 秒时间戳

// ===== 格式化 =====
DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
String str = dt.format(formatter);
LocalDateTime parsed = LocalDateTime.parse("2024-03-15 14:30:00", formatter);

8.3 新旧 API 转换

// Date ↔ Instant ↔ LocalDateTime

// Date → LocalDateTime
LocalDateTime dt = new Date().toInstant()
    .atZone(ZoneId.systemDefault())
    .toLocalDateTime();

// LocalDateTime → Date
Date date = Date.from(LocalDateTime.now()
    .atZone(ZoneId.systemDefault())
    .toInstant());

8.4 SimpleDateFormat vs DateTimeFormatter

SimpleDateFormat

❌ 线程不安全（多线程共享一个实例会出诡异错误）
❌ 可变对象（setXxx 方法修改内部状态）
❌ 异常设计糟糕

DateTimeFormatter

✅ 线程安全（不可变对象）
✅ 支持更丰富的格式
✅ 解析更严格、错误信息更好

不要再用 new Date() 了！日常开发全部用 java.time 包。只有和老框架（MyBatis 旧版等）交互时做转换。

九、CompletableFuture

CompletableFuture 是 JDK 8 引入的异步编程利器，比传统的 Future 多了链式调用、组合编排、异常处理能力。

详见《Java并发编程从原理到实战》第九章的完整讲解。这里补充两个常用模式：

// ===== 异步链式调用 =====
CompletableFuture
    .supplyAsync(() -> fetchUser(userId))           // ① 异步获取用户
    .thenApply(user -> enrichUser(user))              // ② 转换为富用户对象
    .thenAccept(user -> saveToCache(user))              // ③ 写入缓存（不需要返回值）
    .exceptionally(ex -> {                             // ④ 异常兜底
        log.error("failed", ex);
        return null;
    });

// ===== 并行调用 + 结果聚合 =====
CompletableFuture<UserInfo> userF  = CompletableFuture.supplyAsync(() -> queryUser(id));
CompletableFuture<List<Order>> orderF = CompletableFuture.supplyAsync(() -> queryOrders(id));

userF.thenCombine(orderF, (user, orders) -> {       // 合并两个结果
    user.setOrders(orders);
    return user;
}).get(3000, TimeUnit.MILLISECONDS);                  // 超时控制

10.5 StampedLock —— 读写锁的升级版

// 比 ReentrantReadWriteLock 性能更好（乐观读模式）
StampedLock lock = new StampedLock();

long stamp = lock.tryOptimisticRead();  // 乐观读，不阻塞写
// ... 读取数据 ...
if (!lock.validate(stamp)) {           // 检查读期间有没有写
    stamp = lock.readLock();              // 有写 → 升级为悲观读
    try { /* 重新读 */ } finally { lock.unlockRead(stamp); }
}

十一、面试速记

Lambda 表达式的本质是什么？和匿名内部类有什么区别？
Lambda 是函数式接口的实例，编译后不会生成单独的 .class 文件（用 invokedynamic 实现），比匿名内部类更轻量。Lambda 中的 this 指向外层类，匿名内部类的 this 指向内部类自身。
map 和 flatMap 的区别？
map 是一对一映射：Stream<T> → Stream<R>。flatMap 是一对多映射 + 扁平化：Stream<T> → Stream<R>，多个子流合并成一个大流。典型场景：拆分字符串、展开嵌套集合。
Stream 的中间操作和终端操作有什么区别？
中间操作（filter、map、sorted...）惰性求值，返回 Stream 可链式调用，不触发计算。终端操作（collect、forEach、reduce...）触发计算，执行后流关闭。
orElse 和 orElseGet 的区别？
orElse 的参数无论 Optional 是否有值都会执行；orElseGet 的 Supplier 只在无值时才执行。如果默认值构造有副作用或开销大，必须用 orElseGet。
Java 8 的日期 API 比旧 Date/Calendar 好在哪？
不可变（线程安全）、API 设计清晰（Date=年月日，Time=时分秒，分开）、月份从 1 开始、DateTimeFormatter 线程安全、支持时区操作更友好。
函数式接口是什么？常见的有哪几个？
只有一个抽象方法的接口，用 @FunctionalInterface 标记。四大核心：Supplier（供给）、Consumer（消费）、Function（转换）、Predicate（断言）。
接口的 default 方法是什么？为什么要有？
default 方法让接口有方法体实现，子类可选覆盖。核心动机：给 Collection 接口加 stream()，如果加抽象方法所有实现类都得改。default 向后兼容。类的方法优先级 > 接口 default。
Stream 的 reduce 怎么用？
reduce(初始值, (累加结果, 当前元素) -> 新累加结果)。例如：stream.reduce(0, (a,b) -> a+b)。无初始值返回 Optional。特殊场景有 sum()/max()/min() 等 shortcut。
Collectors.groupingBy 怎么用？
groupingBy(分类函数, 下游收集器)。一层分组返回 Map<K, List<T>>，多层嵌套可实现 Map<K, Map<K2, List<T>>>。下游收集器可做 counting()、summing()、averaging() 等聚合操作。
Lambda 里为什么只能引用"有效 final"的局部变量？
Lambda 本质是匿名内部类语法糖，局部变量被拷贝到 Lambda 实例中。如果允许修改原变量，拷贝值就不一致了。实例字段和静态字段不受此限制（因为它们通过引用访问）。
