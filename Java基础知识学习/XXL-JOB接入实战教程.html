一、准备什么环境

要跑起来一个最小可用的 `XXL-JOB`，你至少需要两部分：

调度中心 Admin

  • 带 Web 控制台。
  • 保存任务配置、执行记录、调度状态。
  • 通常需要数据库持久化。

业务执行器 Executor

  • 就是你的 Spring Boot 服务。
  • 它向 Admin 注册自己。
  • 真正执行业务方法的是它。
只有 Executor 没有 Admin，不算完整接入；只有 Admin 没有 Executor，也没有业务任务可以执行。

二、整体接入思路

1. 部署 Admin 2. Spring Boot 引入 xxl-job-core 3. 配置 XxlJobSpringExecutor 4. 编写 @XxlJob 方法 5. 启动服务并注册到 Admin 6. 在控制台新建任务并绑定执行器 7. 手动触发 / 定时触发验证执行结果
1
先让 Admin 可访问。没有控制台，后面所有配置都没法做。
2
在 Spring Boot 工程里引入 `xxl-job-core`，注册执行器 Bean。
3
编写 `@XxlJob("handlerName")` 方法，方法名不重要，`handlerName` 才是控制台绑定的关键。
4
到 Admin 控制台里新建“执行器”和“任务”，把任务路由到你的服务。

三、Spring Boot 端配置

3.1 Maven 依赖

<dependency>
    <groupId>com.xuxueli</groupId>
    <artifactId>xxl-job-core</artifactId>
    <version>2.4.1</version>
</dependency>

3.2 `application.yml` 核心配置

server:
  port: 8085

xxl:
  job:
    admin:
      addresses: http://localhost:8080/xxl-job-admin
    accessToken: default_token
    executor:
      appname: xxl-job-demo-executor
      port: 9999
      logpath: ./logs/xxl-job
      logretentiondays: 15

3.3 注册执行器 Bean

@Bean
public XxlJobSpringExecutor xxlJobExecutor() {
    XxlJobSpringExecutor executor = new XxlJobSpringExecutor();
    executor.setAdminAddresses(adminAddresses);
    executor.setAccessToken(accessToken);
    executor.setAppname(appname);
    executor.setPort(port);
    executor.setLogPath(logPath);
    executor.setLogRetentionDays(logRetentionDays);
    return executor;
}
理解这个 Bean 的作用：它就是告诉你的应用“你现在不仅是普通 Spring Boot 服务，还是 XXL-JOB 的一个执行器节点，要主动向调度中心报到并接收调度命令”。

四、编写 `@XxlJob` 任务

@Component
public class XxlJobHandlers {

    @XxlJob("dailyReportJob")
    public void dailyReportJob() {
        String param = XxlJobHelper.getJobParam();
        XxlJobHelper.log("start dailyReportJob, param={0}", param);

        // 执行业务逻辑

        XxlJobHelper.log("dailyReportJob finished");
    }
}

这里最关键的是 @XxlJob("dailyReportJob") 里的字符串。控制台里配置任务时，JobHandler 要填的就是它。

4.1 分片任务示例

@XxlJob("shardingDemoJob")
public void shardingDemoJob() {
    int shardIndex = XxlJobHelper.getShardIndex();
    int shardTotal = XxlJobHelper.getShardTotal();
    XxlJobHelper.log("processing shard {0}/{1}", shardIndex, shardTotal);
}
分片任务非常适合大批量扫描、批量补偿、分区处理这类作业。核心思想是：同一个任务被拆成多个分片，多个执行器各自处理一部分数据。

五、控制台如何配置任务

服务启动后，真正把任务跑起来，还需要去 Admin 控制台做配置。

1
先创建执行器，`AppName` 要与 Spring Boot 配置中的 `xxl.job.executor.appname` 一致。
2
新建任务，选择刚才的执行器。
3
`JobHandler` 填 `dailyReportJob` 或 `shardingDemoJob`。
4
设置 cron 表达式、路由策略、阻塞策略、失败重试次数。
5
先手动执行一次，确认日志正常，再开启定时调度。
配置项 含义 实战建议
执行器 任务要调度到哪个服务 按业务域划分，不要把所有任务混成一个大执行器
JobHandler 执行器里具体调用哪个方法 和 `@XxlJob("...")` 保持一致
Cron 调度时间规则 先用低频 cron 测试，避免错误任务高频触发
路由策略 任务分发到哪个节点 普通任务先用“第一个”或“轮询”，特殊任务再按机器隔离
阻塞策略 上次未结束时新任务怎么处理 核心任务要认真评估串行还是丢弃

六、从 `@Scheduled` 迁移

迁移时，最好的做法不是把业务逻辑重写一遍，而是把“触发方式”和“业务逻辑”拆开。

@Service
public class ReportService {
    public Map<String, Object> generateDailyReport(String triggerType) {
        ...
    }
}

然后：

旧方式

@Scheduled(cron = "...")
public void localReportJob() {
    reportService.generateDailyReport("@Scheduled");
}

新方式

@XxlJob("dailyReportJob")
public void dailyReportJob() {
    reportService.generateDailyReport("XXL-JOB");
}
迁移原则：把真正的业务逻辑沉到 Service，调度层只负责触发。这样你可以同时保留对照实验，也便于后续从一种调度方案切到另一种。

七、常见问题与排查

为什么控制台里看不到执行器？
先检查 Admin 地址是否正确、服务是否启动、执行器 `appname` 是否和控制台一致、端口是否可访问。
为什么任务触发了但业务没执行？
通常先查 `JobHandler` 名字是否匹配，再看执行日志里是否有异常堆栈。
为什么迁移后还是出现重复处理？
因为调度统一不等于业务天然幂等。核心写操作仍要做去重和幂等控制。
是不是接了 XXL-JOB 就不需要 `@Scheduled` 了？
不是。轻量、实例内、非核心任务仍然可以保留 `@Scheduled`。关键在于按场景分层，而不是“一刀切”。
