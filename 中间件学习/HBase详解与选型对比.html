一、HBase 是什么？为什么需要它？

HBase 是 Hadoop 生态中的分布式、面向列的 KV 数据库，底层基于 HDFS 存储。它的设计目标是处理 海量稀疏数据 的随机读写。

HBase 的核心卖点：
· 十亿行 × 百万列级别的数据量
· 毫秒级随机读写（按 RowKey）
· 水平扩展：加机器就能自动均衡
· 稀疏存储：空列不占空间（列式存储的优势）
· 自动故障转移：依赖 HDFS 的多副本

二、LSM-Tree —— HBase 的核心

HBase 快的原因和 Kafka 一样，也是顺序写。但 HBase 用的不是 Kafka 的日志追加，而是 LSM-Tree（Log-Structured Merge Tree）。

LSM-Tree 写入流程： 写入请求 │ ▼ ┌─────────┐ │ MemStore │ ← 内存中的有序结构（跳表/SkipList） │ (内存) │ 写入先写这里，极快 └────┬────┘ │ 满了（默认 128MB）→ flush 到磁盘 ▼ ┌─────────┐ │ StoreFile│ ← 磁盘上的 HFile，不可变的有序文件 │ (磁盘) │ 每个文件都是有序的 │ (HFile) │ └────┬────┘ │ 小文件太多 → Compaction（合并） ▼ ┌──────────────────────┐ │ Compaction │ │ 小文件 → 大文件 │ ← 合并过程：多路归并排序 + 删除过期数据 │ 删除标记 → 真正删除 │ └──────────────────────┘ LSM-Tree 的核心思想： • 写入 = 顺序写（先内存 MemStore，再磁盘 HFile） • 读取 = 先查 MemStore → 再查 BlockCache → 最后查多个 HFile + 合并结果 • 后台 Compaction 持续合并文件、清理过期数据

B+Tree vs LSM-Tree

B+Tree (MySQL InnoDB)

写：随机写（寻道+旋转延迟），慢
读：直接定位到数据页，快
空间：页分裂浪费空间
适用：读多写少、事务场景

LSM-Tree (HBase / RocksDB)

写：顺序写（追加），极快
读：需要查多个文件合并，慢
空间：Compaction 压缩，紧凑
适用：写多读少、海量数据

三、HBase 核心架构

HBase 架构： ┌──────────┐ │ Client │ └────┬─────┘ │ ① 先问 ZooKeeper：meta 表在哪？ ▼ ┌──────────┐ ② 读 meta 表：我的 RowKey 在哪个 Region？ │ZooKeeper │ ← 集群协调者 └──────────┘ │ ▼ ┌──────────────────────────────────────────┐ │ HMaster │ ← 管理节点 │ (Region 分配、负载均衡、故障恢复) │ └──────────────────────────────────────────┘ │ ▼ ┌──────────────────┐ ┌──────────────────┐ │ RegionServer-1 │ │ RegionServer-2 │ ← 工作节点 │ ┌──────┐┌──────┐ │ │ ┌──────┐┌──────┐ │ │ │Region││Region│ │ │ │Region││Region│ │ ← Region = 一张表的某一段 RowKey 范围 │ │[a,m) ││[m,z) │ │ │ │[0,5) ││[5,9) │ │ │ └──────┘└──────┘ │ │ └──────┘└──────┘ │ └──────────────────┘ └──────────────────┘ │ │ ▼ ▼ ┌──────────────────────────────────────────┐ │ HDFS (DataNode) │ ← 底层存储 └──────────────────────────────────────────┘ Region 是 HBase 的最小分布式单元： • 一张表的数据按 RowKey 范围切分成多个 Region • Region 太大（默认 10GB）→ 自动分裂成两个 • 每个 RegionServer 管理多个 Region

四、RowKey 设计 —— 最重要的事情

RowKey 设计决定了 HBase 的性能和可用性。设计得不好，热点写入、查询慢、扫描效率低全会找上门。

三大原则

原则说明反例
长度不宜过长RowKey 越长，索引开销越大user_behavior_event_2024_01_15_xxx
散列性（避免热点）不要让所有写入都落在一个 Region 上用时间戳做 RowKey → 所有写入打到最后一个 Region
查询友好RowKey 按字典序排列，相近的 RowKey 存一起完全随机 RowKey → 查询只能全表扫

RowKey 设计经典方案

// 场景：智能客服的对话记录表
// 查询需求：按用户 + 时间范围 查询历史对话

// ❌ 错误设计：直接用时间戳
RowKey = timestamp  // "1710316800000" → 热点写在最后一个 Region

// ❌ 错误设计：直接用 user_id
RowKey = user_id    // "user_123" → 如果单个用户数据太多，Region 不均衡

// ✅ 正确设计：散列前缀 + user_id + 反向时间戳
// 反向时间戳 = Long.MAX_VALUE - timestamp, 让新数据排前面
RowKey = MD5(user_id).substring(0, 4) + "_" + user_id + "_" + (Long.MAX_VALUE - timestamp)
// "a3f1_user_123_9223370399469951616"

// 好处：同一个用户的对话在一起（连续扫描快），新数据在前面，散列前缀防热点

五、HBase vs MySQL 分库分表

维度HBaseMySQL 分库分表
数据量百亿行级，天然支撑亿行级开始吃力，需要中间件
横向扩展加机器自动 Rebalance加机器需手动迁移数据
Schema 变更动态列，随时加列DDL 变更，大表锁表风险
SQL 支持不支持 SQL（Phoenix 可提供）完整 SQL 支持
事务不支持跨行事务ACID 事务
复杂查询只支持 RowKey 查 + 范围扫多条件查询、JOIN、子查询
写入性能顺序写，极高B+Tree 随机写，较低
运维复杂度高（依赖 Hadoop/HDFS/ZK）相对简单（ShardingSphere/MyCat）
核心区别：HBase 是"牺牲了复杂查询能力，换来海量数据的快速读写和水平扩展"。MySQL 分库分表是"在不牺牲 SQL 能力的前提下，强行水平扩展"，治标不治本。
如果你的数据查询模式就是按 RowKey 查 + 范围扫，HBase 完美适配。如果需要复杂 SQL，要么用 Phoenix on HBase，要么用 TiDB。

六、HBase vs Cassandra

维度HBaseCassandra
设计理念CP（一致性 + 分区容错）AP（可用性 + 分区容错，最终一致）
存储引擎HDFS 上的 HFile自研存储引擎（SSTable）
依赖HDFS + ZooKeeper无外部依赖，自包含
一致性强一致性（单 Region 内）可调一致性（ONE/QUORUM/ALL）
可用性RegionServer 挂了需恢复（秒级）无单点，任何节点可读写
CQL不支持原生 CQL（类 SQL）
运维重（需维护 Hadoop 生态）轻（无外部依赖）
适用已有 Hadoop 生态、需要强一致性多数据中心、高可用优先

七、HBase vs MongoDB

维度HBaseMongoDB
数据模型宽表（列族 + 动态列）JSON 文档（嵌套结构）
查询能力只支持 RowKey 查询丰富的查询语言 + 聚合管道
写入性能极高高（WiredTiger 引擎）
单机性能需要集群单机也能良好运行
使用门槛高低（JSON 直觉型）
海量数据百亿行级亿级以内（超出需分片集群）
典型场景时序数据、用户行为、日志存储内容管理、用户画像、实时分析

八、选型决策树

海量数据存储选型： 你的需求是？ ├── 需要 SQL（JOIN、子查询、事务）→ MySQL 分库分表 / TiDB ├── 数据量大(十亿+)、按 ID 查 + 范围扫 → HBase │ └── 不想维护 Hadoop → Cassandra 或 ScyllaDB ├── 灵活的文档模型、JSON 嵌套数据 → MongoDB ├── 时序数据（监控、IoT）→ HBase / InfluxDB / TDengine └── 智能客服场景中的对话存储： ├── 按 user_id + 时间范围查对话历史 → HBase 最合适 │ └── RowKey = hash(user_id) + user_id + reverse_timestamp └── 数据量不大(<千万级) → MySQL 就够了，别过度设计

九、HBase 基础操作

# ===== HBase Shell 基础操作 =====
# 创建表（列族 cf）
create 'conversation', 'cf'

# 插入数据
put 'conversation', 'rowkey_001', 'cf:user_id', 'user_123'
put 'conversation', 'rowkey_001', 'cf:question', '怎么退款？'
put 'conversation', 'rowkey_001', 'cf:answer', '请打开订单...'

# 按 RowKey 查询
get 'conversation', 'rowkey_001'

# 范围扫描（按 RowKey 范围）
scan 'conversation', {STARTROW => 'a3f1_user_123_', STOPROW => 'a3f1_user_123|'}

# 删除
delete 'conversation', 'rowkey_001', 'cf:question'
# 删除整行
deleteall 'conversation', 'rowkey_001'

// 创建连接
Configuration config = HBaseConfiguration.create();
config.set("hbase.zookeeper.quorum", "localhost:2181");
Connection connection = ConnectionFactory.createConnection(config);
Table table = connection.getTable(TableName.valueOf("conversation"));

// 插入
Put put = new Put(Bytes.toBytes("rowkey_001"));
put.addColumn(Bytes.toBytes("cf"), Bytes.toBytes("question"), Bytes.toBytes("怎么退款？"));

table.put(put);

// 按 RowKey 查询

ResultScanner scanner = table.getScanner(scan);

十、面试速记

HBase 的 LSM-Tree 和 MySQL 的 B+Tree 有什么区别？
LSM-Tree 写入顺序追加（先 MemStore 内存 → HFile 磁盘），写极快但读要合并多个文件。B+Tree 写入要随机 I/O（页分裂），写慢但读直接定位。HBase 适合写多读少、海量数据；MySQL 适合读多写少、事务场景。
HBase 的 Region 是什么？怎么分裂的？
Region 是一张表某一段 RowKey 范围的数据集合（默认 10GB）。Region 太大 → 分裂成两个子 Region → 均衡分布到不同 RegionServer。Region 是 HBase 的最小分布式和负载均衡单元。
RowKey 设计有什么讲究？
长度不宜过长、必须有散列性（防止热点）、查询友好（相邻 RowKey 存一起）。经典设计：hash(user_id)前4位 + user_id + reverse_timestamp。
HBase 和 MySQL 分库分表怎么选？
HBase 天然支持百亿级数据、动态列、水平扩展，但只支持 RowKey 查询。MySQL 分库分表保留 SQL 能力但运维复杂。按 RowKey 查为主 → HBase；需要 JOIN/子查询 → MySQL/TiDB。
HBase 为什么能实现毫秒级读写？
写入：先写 MemStore（内存）→ 极快；刷盘写 HFile → 顺序写 → 快。读取：BlockCache（LRU 缓存热数据）+ Bloom Filter（快速判断数据是否在 HFile 中，跳过不相关的文件）。
HBase vs Cassandra 选谁？
需要强一致性 + 已有 Hadoop 生态 → HBase。需要高可用 + 无单点 + 多数据中心 + 不想维护 Hadoop → Cassandra。HBase 是 CP，Cassandra 是 AP。
