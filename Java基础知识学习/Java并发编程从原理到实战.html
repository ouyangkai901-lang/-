一、并发编程基础

1.1 为什么需要并发？

单线程程序在等待 IO（磁盘、网络）时会白白浪费 CPU 资源。并发编程让多个任务交替执行， CPU 不必等待，整体吞吐量大幅提升。但代价是 复杂度上升：原子性、可见性、有序性三大问题。

三大核心问题：
· 原子性：一个或多个操作，要么全部执行完毕不被中断，要么全不执行
· 可见性：线程 A 修改了共享变量，线程 B 是否能立即看到
· 有序性：编译器/CPU 可能会对指令重排序，多线程下可能出现意外结果

1.2 Java 内存模型（JMM）—— 理解"可见性"的关键

JMM 规定：每个线程有自己的 工作内存（L1/L2 缓存），主内存是共享的。线程修改变量后必须刷新到主内存，其他线程才能看到。

Java 内存模型的"8 种原子操作"： lock (锁定主内存变量) │ ▼ read (从主内存读取 → 工作内存) │ ▼ load (把 read 到的值 → 工作内存副本) │ ▼ use (线程执行引擎使用该值) │ ▼ assign(执行引擎赋值 → 工作内存副本) │ ▼ store (工作内存副本 → 传给主内存) │ ▼ write (写入主内存变量) │ ▼ unlock(解锁)

happens-before 规则是 JMM 中最核心的概念：如果操作 A happens-before 操作 B，则 A 的结果对 B 可见。几条最重要的规则：

规则含义
程序次序规则同一线程内，前面的操作 happens-before 后面的操作
volatile 规则volatile 写 happens-before 后续的 volatile 读
锁规则解锁 happens-before 后续对同一锁的加锁
传递性如果 A hb B, B hb C → A hb C
线程启动规则Thread.start() hb 该线程内的任何操作

二、synchronized —— 从偏向锁到重量级锁

2.1 基础用法

// ① 修饰实例方法 —— 锁是 this
public synchronized void doSomething() { ... }

// ② 修饰静态方法 —— 锁是 Class 对象

// ③ 修饰代码块 —— 锁是指定对象

2.2 锁升级（膨胀）过程 —— 面试核心！

JDK 1.6 之后，synchronized 不再是"直接拿重量级锁"，而是 逐步升级 的锁膨胀机制：

锁升级过程： 无锁 ──▶ 偏向锁 ──▶ 轻量级锁 ──▶ 重量级锁 │ (第一个线程获取锁) MarkWord 记录线程ID │ ▼ (另一个线程竞争) 检查占用线程是否存活 不存活 → 撤销偏向，恢复无锁 存活 → 升级到轻量级锁 │ ▼ (CAS 自旋获取锁) 自旋成功 → 持锁运行 自旋失败 (超过10次) → 升级到重量级锁 │ ▼ (操作系统互斥量 mutex) 未拿到锁的线程 → BLOCKED 状态 → 内核态等待

MarkWord 是锁升级的载体

每个 Java 对象的对象头（Object Header）中都有一个 Mark Word（32位/64位），它存储了对象的 hashCode、GC 分代年龄和锁状态。

锁状态Mark Word 内容是否偏向锁标志位
无锁对象 hashCode + 分代年龄001
偏向锁线程 ID + Epoch + 分代年龄101
轻量级锁指向栈中锁记录（Lock Record）的指针—00
重量级锁指向 monitor （互斥量）的指针—10
GC 标记空—11
为什么设计锁升级？
大多数时候锁根本不存在竞争，甚至总是 同一个线程 反复获取同一把锁。 如果每次都走重量级锁（操作系统 mutex），线程阻塞和唤醒需要用户态↔内核态切换，成本极高。
偏向锁和轻量级锁全程在用户态完成，只有真正出现严重竞争才升级到重量级锁。

2.3 synchronized 与 ReentrantLock 对比

synchronized（内置锁）

✅ 语法简单，自动释放
✅ JVM 层面优化（锁升级）
❌ 不可打断，一直等
❌ 不支持公平锁
❌ 不支持多个 Condition
❌ 拿不到锁就阻塞，没有尝试/超时机制

ReentrantLock（显式锁）

✅ 可打断：lockInterruptibly()
✅ 可超时：tryLock(timeout)
✅ 支持公平锁
✅ 支持多个 Condition
✅ 灵活：非阻塞尝试 tryLock()
❌ 必须手动释放（finally unlock）

三、volatile —— 可见性与有序性

3.1 volatile 保证什么？不保证什么？

能力保证？说明
可见性✅ 保证写 volatile 变量后立即刷新到主内存；读 volatile 变量时从主内存读取最新值
有序性✅ 保证通过内存屏障禁止指令重排序：写之前不重排，读之后不重排
原子性❌ 不保证volatile int i = 0; i++ 不是原子操作（读→改→写，三步）

3.2 经典场景：DCL 单例为什么必须 volatile

public class Singleton {
    // ★★★ 这里的 volatile 绝对不能省！★★★
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {                        // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {                    // 第二次检查
                    instance = new Singleton();        // 问题在这里！
                }
            }
        }
        return instance;
    }
}

问题所在：instance = new Singleton() 这行代码实际包含了三步：

分配内存：JVM 为 Singleton 对象分配堆空间
初始化对象：调用构造函数，初始化成员变量
引用赋值：将 instance 指向分配的内存地址

CPU 可能对步骤 2 和 3 重排序：先赋值引用，再初始化。如果线程 A 刚执行完步骤 3（还未执行步骤 2），线程 B 第一次检查 instance != null，直接返回了一个尚未初始化完毕的对象，使用时就会出现空指针等奇怪问题。

volatile 的作用：通过 StoreStore + StoreLoad 内存屏障，禁止步骤 2 和 3 重排序，保证线程 B 看到的 instance 一定是已经初始化完成的完整对象。

3.3 内存屏障（Memory Barrier）

屏障类型作用
LoadLoad屏障前的 Load 必须先于屏障后的 Load 完成
StoreStore屏障前的 Store 必须先于屏障后的 Store 完成
LoadStore屏障前的 Load 必须先于屏障后的 Store 完成
StoreLoad屏障前的 Store 必须先于屏障后的 Load 完成（最强屏障，同时具备其他三种效果）

volatile 写时JVM 插入 StoreStore + StoreLoad 屏障；volatile 读时JVM 插入 LoadLoad + LoadStore 屏障。

四、AQS —— J.U.C 的核心骨架

AQS（AbstractQueuedSynchronizer）是整个 J.U.C 包的底层框架。ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 全部基于 AQS 构建。

4.1 核心设计

AQS 的本质是一个 FIFO 双向队列 + 一个 int 状态变量：

AQS 数据结构： state (volatile int) ← 同步状态，CAS 修改 │ ▼ ┌───────┐ ┌───────┐ ┌───────┐ │ Node │ → │ Node │ → │ Node │ ← CLH 队列 (FIFO 双向链表) │ 线程1 │ │ 线程2 │ │ 线程3 │ │ 等待中 │ │ 等待中 │ │ 等待中 │ └───────┘ └───────┘ └───────┘ head tail

state 是核心。子类通过控制 state 的获取和释放来实现不同的同步语义：

实现类state 含义
ReentrantLock0=未锁，1=已锁，n=重入次数（同一个线程）
Semaphore剩余许可数量
CountDownLatch倒计数值，count=0 时唤醒等待线程
ReentrantReadWriteLock高 16 位=读锁持有次数，低 16 位=写锁持有次数

4.2 获取锁的流程

// ReentrantLock 内部使用 Sync（继承 AQS）
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&                       // ① 子类实现，CAS 尝试获取
        acquireQueued(addWaiter(Node.EXCLUSIVE), arg)) // ② 失败→入队→自旋/阻塞
        selfInterrupt();                             // ③ 被唤醒后补上中断标志
}
获取锁流程（以 ReentrantLock 为例）： ① tryAcquire(arg): 检查 state==0 ? ├── 是 → CAS 设 state=1，设置 owner 为当前线程 → 成功 └── 否 → state>0 且 owner 是自己 → state++（重入） → owner 是别人 → 返回 false，继续 ↓ ② addWaiter(Node.EXCLUSIVE): 当前线程封装成 Node，CAS 入队尾 ③ acquireQueued(node): 自旋 ├── 前驱是 head → 再 tryAcquire 一次 │ ├── 成功 → 出队，成为新 head │ └── 失败 → 继续 └── 前驱不是 head → park() 挂起（Unsafe.park） └── 前驱释放时 unpark 唤醒 → 继续自旋

4.3 释放锁的流程

public final boolean release(int arg) {
    if (tryRelease(arg)) {          // ① 子类实现，state-- 或设为 0
        Node h = head;
        if (h != null && h.waitStatus != 0)
            unparkSuccessor(h);        // ② 唤醒 head 的后继节点
        return true;
    }
    return false;
}

// ReentrantLock 的 tryRelease：
protected final boolean tryRelease(int releases) {
    int c = getState() - releases;      // state - 1
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    boolean free = false;
    if (c == 0) {
        free = true;
        setExclusiveOwnerThread(null);    // 重入全部释放才清 owner
    }
    setState(c);                         // CAS 设置 state
    return free;
}

五、ReentrantLock —— 可重入的显式锁

5.1 公平锁 vs 非公平锁

非公平锁（默认）NonfairSync

// 一上来直接 CAS 抢锁
final void lock() {
    if (compareAndSetState(0, 1))
        setExclusiveOwnerThread(...);
    else
        acquire(1);  // 抢失败再排队
}

性能更好：刚释放锁的线程可能仍在 CPU 上，再获取成功率高，减少上下文切换

公平锁 FairSync

// 先检查队列
final void lock() {
    acquire(1);  // 走排队流程
}
protected final boolean tryAcquire(...) {
    if (state == 0) {
        // hasQueuedPredecessors(): 前面有人在排吗？
        if (!hasQueuedPredecessors() && ...)
            ... // 只有队首才能抢
    }
}

线程饥饿概率低，但频繁上下文切换，吞吐量差

5.2 Condition —— 等待/通知机制

ReentrantLock 可以创建多个 Condition，实现精确唤醒。而 synchronized 只有一个等待队列（wait/notify）。

class BoundedBuffer {
    final ReentrantLock lock = new ReentrantLock();
    final Condition notFull  = lock.newCondition();  // 缓冲区不满
    final Condition notEmpty = lock.newCondition();  // 缓冲区不空

    final Object[] items = new Object[100];
    int putptr, takeptr, count;

    public void put(Object x) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length)
                notFull.await();           // 满了，等 notFull 信号
            items[putptr] = x;
            if (++putptr == items.length) putptr = 0;
            ++count;
            notEmpty.signal();              // 有数据了，唤醒消费者
        } finally { lock.unlock(); }
    }

    public Object take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await();          // 空了，等 notEmpty 信号
            Object x = items[takeptr];
            if (++takeptr == items.length) takeptr = 0;
            --count;
            notFull.signal();               // 有空位了，唤醒生产者
            return x;
        } finally { lock.unlock(); }
    }
}

六、线程池 —— 7 大参数与调优

6.1 为什么要用线程池？

复用线程，减少创建/销毁开销（线程的创建约 1ms，约 1MB 栈内存）；控制最大并发数，防止无限制创建线程导致 OOM。

6.2 核心构造函数：7 大参数

public ThreadPoolExecutor(
    int corePoolSize,        // ① 核心线程数
    int maximumPoolSize,     // ② 最大线程数
    long keepAliveTime,      // ③ 空闲线程存活时间
    TimeUnit unit,             // ④ 时间单位
    BlockingQueue<Runnable> workQueue, // ⑤ 任务队列（阻塞队列）
    ThreadFactory threadFactory,      // ⑥ 线程工厂（命名/守护）
    RejectedExecutionHandler handler   // ⑦ 拒绝策略
)

6.3 线程池工作流程（面试必画！）

任务提交后的执行流程： task 到达 │ ▼ 核心线程数 < corePoolSize？ ├── 是 → 创建新核心线程执行任务 │ └── 否 → 工作队列满了？ │ ├── 否 → 放入工作队列等待 │ └── 是 → 线程数 < maximumPoolSize？ │ ├── 是 → 创建新非核心线程执行 │ └── 否 → 执行 拒绝策略

6.4 4 种拒绝策略

策略行为适用场景
AbortPolicy（默认）抛出 RejectedExecutionException必须感知拒绝的场景
CallerRunsPolicy由提交任务的线程（调用者）自己执行主线程能分担，可配合限流
DiscardPolicy直接丢弃，不抛异常允许丢任务（如日志采样）
DiscardOldestPolicy丢弃队列头（最旧）的任务，重新提交新任务比旧任务更重要

6.5 参数调优：CPU 密集 vs IO 密集

CPU 密集型

大部分时间在 计算（加密、图像处理、算法）

corePoolSize = CPU 核数 + 1
// 加 1 是为了偶尔的缺页中断

线程再多也没用，反而因上下文切换降低效率

IO 密集型

大部分时间在 等待 IO（网络请求、数据库查询）

corePoolSize = CPU 核数 × 2
// 或者：
= CPU 核数 × (1 + 等待时间/计算时间)

线程在等 IO 时 CPU 空闲，多开线程提高吞吐

6.6 常见线程池（Executors —— 不建议直接使用）

工厂方法队列风险
newFixedThreadPool(n)无界 LinkedBlockingQueue队列无限长 → 任务堆积 → OOM
newCachedThreadPool()SynchronousQueue（不存任务，直接交线程）最大线程数 Integer.MAX_VALUE → OOM
newSingleThreadExecutor()无界 LinkedBlockingQueue同 FixedThreadPool(1)，OOM
阿里开发规范强制：必须用 new ThreadPoolExecutor() 显式传参，不允许用 Executors 工厂方法！
最佳实践：
· ThreadFactory 一定要给线程命名，方便监控排查
· 队列建议用有界的 ArrayBlockingQueue，配合 CallerRunsPolicy 做背压
· 如果需要任务完成后回调，使用 submit() 返回 Future

七、J.U.C 工具类

7.1 CountDownLatch —— 一等多

一个线程等待多个线程完成任务。计数器减到 0 时，await 的线程被唤醒。不可复用，用一次就没。

// 场景：裁判等 3 个运动员都到达终点
CountDownLatch latch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown();      // 每个运动员到达 → count--
    }).start();
}
latch.await();                     // 裁判等待 count=0
// 所有人到达，继续执行

7.2 CyclicBarrier —— 多对多互相等

一组线程互相等待，所有人都到达屏障点后再一起继续。可循环复用。

// 场景：3 个选手准备好后同时起跑
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    // 所有人都到齐后执行的回调（由最后一个到达的线程执行）
    System.out.println("所有人就位，出发！");
});

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        prepare();
        barrier.await();        // 等人齐
        run();                  // 人齐了一起跑
    }).start();
}

7.3 Semaphore —— 信号量，控制并发数量

// 场景：数据库连接池最多 5 个连接
Semaphore semaphore = new Semaphore(5);  // 5 个许可

for (int i = 0; i < 20; i++) {
    new Thread(() -> {
        semaphore.acquire();          // 获取许可（阻塞）
        try {
            useDBConnection();   // 最多 5 个线程同时执行
        } finally {
            semaphore.release();      // 释放许可
        }
    }).start();
}

7.5 Atomic 原子类

基于 CAS（Compare And Swap） 实现的无锁原子操作。比 synchronized 轻量，适合简单的计数/标记场景。

// CAS 的三个操作数：内存地址 V、期望旧值 A、新值 B
// 如果 V == A，则更新为 B，返回 true
// 否则不更新，返回 false，自旋重试

public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);          // 读取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS
    return v;
}
类型原子类特点
基本类型AtomicInteger, AtomicLong, AtomicBoolean单个值的原子操作
数组AtomicIntegerArray, AtomicLongArray数组元素的原子更新
引用AtomicReference对象引用的原子 CAS
字段更新AtomicIntegerFieldUpdater原子更新对象中的 volatile 字段
累加器LongAdder, DoubleAdder高并发下比 AtomicLong 性能更好（分 Cell 分散竞争）
CAS 的问题：
· ABA 问题：值从 A→B→A，CAS 检查认为是 A 不变 — 解决：AtomicStampedReference 带版本号
· 自旋 CPU 开销：CAS 失败会一直循环重试 — 通常自旋次数有限，超限则用锁
· 只能保证一个变量的原子操作 — 多个变量需要 AtomicReference 或锁

        if (tab == null)
            tab = initTable();             // 2. 初始化 table（CAS）
        else if ((f = tabAt(tab, i = (tab.length-1) & hash)) == null) {
            // 3. 桶为空 → CAS 直接插入
            if (casTabAt(tab, i, null, new Node<>(hash, key, value)))
                break;
        }
        else if (f.hash == MOVED)
            tab = helpTransfer(tab, f);      // 4. 正在扩容，帮忙迁移
        else {
            // 5. 锁住桶的头节点，插入链表或红黑树
            synchronized (f) {
                // ... 遍历链表/红黑树，插入或覆盖
            }
        }
    }
}

8.4 ConcurrentHashMap 的 size() 怎么算？

不是锁住整个表去 count，而是：

  • JDK 1.7：统计每个 Segment 的 count（遍历 Segment，尝试 3 次不加锁计算，若期间变化则加锁重算）
  • JDK 1.8：在所有写操作时通过 addCount() 累加到一个 baseCount + CounterCell 数组，读 size 时 sum 所有 CounterCell + baseCount

8.5 JDK 1.8 扩容 —— 多线程协同迁移

扩容时，在旧的 table 上插入 ForwardingNode 标记该桶已经迁移完毕。其他线程在读写时遇到 ForwardingNode 会帮忙迁移（helpTransfer），分散扩容压力。

九、CompletableFuture —— 异步编程

9.1 为什么要用 CompletableFuture？

Future 的痛点：get() 阻塞等待结果 → CPU 闲着。多个 Future 之间有依赖关系 → 写成回调地狱。

CompletableFuture：支持 链式调用、组合编排、异常处理、超时控制，把你的异步任务组合成一棵执行树。

9.2 核心方法速览

类别方法作用
创建supplyAsync(() -> result)有返回值的异步任务
创建runAsync(() -> {})无返回值的异步任务
链式thenApply(fn)拿到上一个结果 → 转换 → 返回新结果
链式thenAccept(consumer)消费结果，无返回值
链式thenRun(runnable)不关心上一步结果，继续执行
组合thenCombine(other, fn)两个任务都完成，合并结果
组合thenCompose(fn)依赖上一个结果，产生新的 CF（扁平化）
并行allOf(cf1, cf2, ...)全部完成才继续
并行anyOf(cf1, cf2, ...)任意一个完成就继续
异常exceptionally(fn)异常时返回兜底值
异常handle((res, ex) -> ...)正常和异常都走这里

9.3 实战：并行调用 + 聚合结果

// 场景：查询用户信息，需要调 3 个微服务
CompletableFuture<UserInfo> userFuture = CompletableFuture
    .supplyAsync(() -> queryUser(userId));        // ① 查用户

CompletableFuture<List<Order>> orderFuture = CompletableFuture
    .supplyAsync(() -> queryOrders(userId));    // ② 查订单

CompletableFuture<Integer> pointFuture = CompletableFuture
    .supplyAsync(() -> queryPoints(userId));    // ③ 查积分

// 三个都完成 → 合并结果
CompletableFuture<UserDTO> result = userFuture
    .thenCombine(orderFuture, (user, orders) -> {    // ④ 合并 ①+②
        user.setOrders(orders);
        return user;
    })
    .thenCombine(pointFuture, (user, points) -> {    // ⑤ 合并 ①+②+③
        user.setPoints(points);
        return user;
    });

// 或者等所有完成
CompletableFuture.allOf(userFuture, orderFuture, pointFuture)
    .join();  // 等待所有完成

UserDTO dto = result.get(3000, TimeUnit.MILLISECONDS);  // 超时控制

9.4 线程池一定要自定义！

supplyAsync(Runnable) 默认使用 ForkJoinPool.commonPool()（CPU 核数 - 1 个线程），多路并行时性能瓶颈明显。
必须：supplyAsync(runnable, myThreadPool) 传入自定义线程池。

十、ThreadLocal

10.1 一个引子：同样一个 ThreadLocal 对象，不同线程为什么拿到不同的值？

先看一段代码，建立一个直观感受：

// 全局只有一个 ThreadLocal 对象！
public static final ThreadLocal<String> tl = new ThreadLocal<>();

new Thread(() -> {
    tl.set("小明");
    System.out.println(tl.get());   // → "小明"
}).start();

new Thread(() -> {
    tl.set("小红");
    System.out.println(tl.get());   // → "小红"
}).start();

关键点：全局只有一个 tl 对象，但线程1和线程2各自 set 之后，各自的 get() 拿到的是自己 set 的值，互不干扰。这跟普通变量完全不同——同一个对象在不同的线程中竟然存储了不同的值。

10.2 ThreadLocal 与普通变量的本质区别

变量类型作用域线程安全？生命周期典型场景
局部变量
void foo() { int x = 1; }		方法栈帧内，每个线程调用方法时在自己栈上分配✅ 安全方法执行完即销毁临时计算结果
实例变量
class A { int x; }		对象在堆上，所有访问该对象的线程共享同一份❌ 不安全跟着对象走，对象被 GC 才销毁对象的状态属性
静态变量
static int x;		类在方法区，全局唯一，所有线程共享❌ 不安全类加载到卸载（基本=应用生命周期）全局配置、计数器
ThreadLocal
ThreadLocal<T> tl		tl 对象本身是共享的，但 tl 内部通过 Thread → ThreadLocalMap 让每个线程存自己的副本✅ 安全跟着线程走，线程销毁才清理（或手动 remove）数据库连接、Session、Request 上下文
一句话总结：普通变量是"线程去共享内存里读写同一个地址"，ThreadLocal 是"线程随身带了一个 HashMap，ThreadLocal 对象只是 key，值存在自己的 HashMap 里"。

10.3 隔离机制：同一个 ThreadLocal 对象如何在不同线程中存储不同值？

核心原理：每个 Thread 内部都有一个 ThreadLocalMap，ThreadLocal 对象只充当这个 Map 的 key。

完整的 ThreadLocal 数据结构： ThreadLocal tl Thread-1 Thread-2 (全局唯一对象) ┌─────────────────┐ ┌─────────────────┐ │ │ threadLocals │ │ threadLocals │ │ │ (ThreadLocalMap) │ │ (ThreadLocalMap)│ │ │ ┌──────┬──────┐ │ │ ┌──────┬──────┐ │ ├────────────────────→││Entry0│"小明"│ │ │ │Entry0│"小红"│ │ │ │ │key=tl│value │ │ │ │key=tl│value │ │ │ │ └──────┴──────┘ │ │ └──────┴──────┘ │ │ └─────────────────┘ └─────────────────┘ 关键： • ThreadLocal 对象 tl 本身 → 存储在堆上，所有线程共享 • tl.set(value) → 取得当前线程 → 取得当前线程的 threadLocals → 以 tl 为 key 存 value • tl.get() → 取得当前线程 → 取得当前线程的 threadLocals → 以 tl 为 key 取 value

源码层面：set / get 到底做了什么？

// ThreadLocal.set() 源码（简化）
public void set(T value) {
    Thread t = Thread.currentThread();              // ① 获取当前线程
    ThreadLocalMap map = getMap(t);                  // ② 获取当前线程的 ThreadLocalMap
    if (map != null)
        map.set(this, value);                      // ③ 以 this(ThreadLocal 对象) 为 key 存入
    else
        createMap(t, value);                        // ④ 第一次用，先创建 ThreadLocalMap
}

public T get() {
    Thread t = Thread.currentThread();              // ① 获取当前线程
    ThreadLocalMap map = getMap(t);                  // ② 获取当前线程的 ThreadLocalMap
    if (map != null) {
        ThreadLocalMap.Entry e = map.getEntry(this); // ③ 以 this 为 key 查找
        if (e != null)
            return (T) e.value;                   // ④ 返回找到的 value
    }
    return setInitialValue();                     // ⑤ 没找到，返回初始值
}

// getMap 极其简单：直接返回 Thread 对象的 threadLocals 字段
ThreadLocalMap getMap(Thread t) {
    return t.threadLocals;   // 每个线程自己持有的 Map
}
所以隔离的秘密就在 Thread.currentThread()：
· 同一个 ThreadLocal.set(value) 被线程1调用 → currentThread()=线程1 → 存进线程1的 Map
· 同一个 ThreadLocal.set(value) 被线程2调用 → currentThread()=线程2 → 存进线程2的 Map
· 存的是不同的 Map，自然互不影响！

10.4 内部实现：ThreadLocalMap 的数据结构

ThreadLocalMap 不是用 JDK 的标准 HashMap，而是自己实现的一个简易 Map：

ThreadLocalMap 结构： 内部使用 Entry[] 数组（初始容量 16，扩容阈值 2/3） Entry 继承自 WeakReference： ┌────────────────────────────────┐ │ Entry │ │ • referent → ThreadLocal 对象 │ ← 弱引用（key） │ • value → 线程私有的数据 │ ← 强引用 └────────────────────────────────┘ 解决 hash 冲突的方式：开放地址法（线性探测） 冲突后 index = (index + 1) % len，直到找到空位 （不同于 HashMap 的链表法/红黑树）

10.5 内存泄漏问题 —— 深度剖析

第一步：理解"为什么 key 要用弱引用"

假如 key 是强引用（即 Entry 持有的不是 WeakReference，而是直接引用 ThreadLocal 对象）：

假设 key 是强引用： 代码： public void handleRequest() { ThreadLocal<Context> tl = new ThreadLocal<>(); tl.set(context); // 业务逻辑... } // ← 方法结束，局部变量 tl 销毁 方法结束后： 栈上的 tl 引用 → 销毁 ✅ 堆上的 ThreadLocal 对象 ← 还在！被 Entry（强引用）引用着 → ThreadLocal 永远无法被 GC → 泄漏！ 如果每次请求 new 一个 ThreadLocal，又不 remove → 泄漏日积月累

所以 key 用弱引用的好处：当 ThreadLocal 对象的外部强引用消失后，GC 至少能回收 key（即 ThreadLocal 对象本身），不会导致 ThreadLocal 对象无限堆积。

第二步：但光回收 key 还不够 —— value 仍然泄漏

这就是 ThreadLocal 内存泄漏的核心矛盾：

内存泄漏的完整链条： 阶段 ①：正常使用 ┌─────────────┐ │ ThreadLocalMap│ │ ┌───────────┐ │ │ │ Entry │ │ │ │ key → [TL对象] ← 外部强引用（栈上的 tl 变量） │ │ value→ [数据] │ │ └───────────┘ │ └─────────────┘ 阶段 ②：外部强引用断开（tl = null 或方法结束） ┌─────────────┐ │ ThreadLocalMap│ │ ┌───────────┐ │ │ │ Entry │ │ │ │ key → [TL对象] ← 只剩弱引用！GC 下次就回收 │ │ value→ [数据] │ │ └───────────┘ │ └─────────────┘ 阶段 ③：GC 发生后 ┌─────────────┐ │ ThreadLocalMap│ │ ┌───────────┐ │ │ │ Entry │ │ │ │ key → null │ ← key 被回收了 │ │ value→ [数据] │ ← 但 value 依然被 Entry 强引用！无法回收！ │ └───────────┘ │ └─────────────┘ 阶段 ④：线程池场景 —— 致命一击 普通线程：线程执行完 → 销毁 → ThreadLocalMap 被回收 → value 跟着回收 ✅ 线程池线程：线程一直存活 → ThreadLocalMap 一直存在 → key=null 的 Entry 越来越多 → 内存持续增长 → OOM
所以 ThreadLocal 内存泄漏的根因是：
1. Entry 对 key 是弱引用，对 value 是强引用
2. key（ThreadLocal 对象）可以被 GC，但 value 不行
3. key 被 GC 后变成 null，value 永远无法通过 key 被访问到，但也无法被回收
4. 线程池场景下线程不销毁，这些"僵尸" Entry 持续累积

第三步：JDK 的"自救"机制 —— 惰性清理

JDK 也意识到了这个问题，在 ThreadLocalMap 的 set()、get()、remove() 方法中会触发清理：

// ThreadLocalMap.getEntry() 内部（简化）
private Entry getEntry(ThreadLocal<?> key) {
    int i = key.threadLocalHashCode & (table.length - 1);
    Entry e = table[i];
    if (e != null && e.get() == key)
        return e;                           // 命中
    else
        return getEntryAfterMiss(key, i, e); // 未命中 → 触发清理
}

// getEntryAfterMiss 在遍历过程中遇到 key==null 的 Entry
// 会调用 expungeStaleEntry(i) 清除它：
//   1. entry.value = null     → 解除对 value 的强引用
//   2. table[i] = null        → 清除 Entry 本身
但惰性清理并不可靠：
只有后续有线程继续调用该 ThreadLocalMap 的 set/get/remove 时才会触发清理。
如果某个 ThreadLocal 被废弃后，对应的线程长期没有使用任何 ThreadLocal → 僵尸 Entry 就一直堆着。所以别指望 JDK 帮你全搞定，自己 remove 才是正道。

第四步：最佳实践

// ★ 正确用法一：try-finally 保证 remove
ThreadLocal<UserContext> context = new ThreadLocal<>();
try {
    context.set(new UserContext(userId));
    // 执行业务逻辑
} finally {
    context.remove();  // ★★★ 必须 remove！★★★
}

// ★ 正确用法二：JDK 21 开始用 AutoCloseable（如果有预览特性）
// ★ 正确用法三：不要用 static ThreadLocal
//   static 变量生命周期 = 应用生命周期 → 永远有强引用 → remove 也可能忘

// ★ 正确用法四：如果必须用 static，一定要确保 remove
public class RequestContextHolder {
    private static final ThreadLocal<Context> holder = new ThreadLocal<>();

    public static void set(Context ctx) { holder.set(ctx); }
    public static Context get() { return holder.get(); }
    public static void clear() { holder.remove(); }  // ← 暴露 remove 方法
}

// 在 Filter/Interceptor 中 finally 调用 clear()

十二、阿里面试真题速记

synchronized 锁升级过程？
无锁 → 偏向锁（同一线程）→ 轻量级锁（CAS 自旋）→ 重量级锁（OS mutex）。MarkWord 中的锁标志位记录当前状态。偏向锁不释放，下次同一线程来直接获取；轻量级通过 CAS 自旋，失败 10 次升级。
AQS 原理？
volatile int state + FIFO 双向队列（CLH 变种）。获取锁时 CAS 改 state，失败后入队自旋/阻塞；释放时唤醒后继节点。ReentrantLock、Semaphore、CountDownLatch 全部基于此。
线程池 7 大参数和 4 种拒绝策略？
corePoolSize / maximumPoolSize / keepAliveTime / unit / workQueue / threadFactory / handler。拒绝：Abort(抛异常)、CallerRuns(调用者执行)、Discard(丢弃)、DiscardOldest(丢弃最旧)。
线程池工作流程？
corePoolSize 满了 → 入队列 → 队列满了 → 创建到 maxPoolSize → maxPoolSize 还满 → 拒绝策略。非核心线程空闲超过 keepAliveTime 则回收。
volatile 保证什么？不保证什么？
保证：可见性（写立即刷主内存）、有序性（内存屏障禁止重排序）。不保证：原子性（i++ 三步不是原子的）。经典场景：DCL 单例（禁止指令重排序）。
ConcurrentHashMap JDK 1.7 vs 1.8？
1.7：Segment 分段锁（16 段，ReentrantLock）；1.8：CAS + synchronized 锁链表头节点（单桶），数据结构同 HashMap（数组+链表+红黑树），读完全无锁，扩容多线程协作。
ThreadLocal 如何实现线程隔离？
每个 Thread 内部有一个 ThreadLocalMap（类似 HashMap）。ThreadLocal.set(value) → 获取当前线程 → 获取其 ThreadLocalMap → 以 this（ThreadLocal 对象）为 key 存入 value。不同线程 set 同一个 TL 对象，写入的是各自线程的 Map，互不干扰。
ThreadLocal 内存泄漏原因？怎么解决？
ThreadLocalMap.Entry 继承 WeakReference，key 是弱引用、value 是强引用。key 被 GC 回收后变为 null，但 value 在 Entry 中仍是强引用，无法回收。线程池线程不销毁，僵尸 Entry 持续累积。JDK 有惰性清理（set/get/remove 时碰见 key=null 会清理），但不可靠。解决：finally 中 remove()。
CountDownLatch vs CyclicBarrier？
Latch = 一等多的门闩，countDown 减到 0 后 await 通过，不可复用。Barrier = 多对多的栅栏，人齐了一起冲，可循环使用。
ReentrantLock 公平锁 vs 非公平锁？
非公平：新来的线程直接 CAS 抢锁，抢不到再排队，吞吐高。公平：严格按队列顺序，先到先得，防止饥饿但上下文切换频繁。默认非公平。
CompletableFuture 的核心方法？
supplyAsync/runAsync 创建；thenApply/thenAccept/thenRun 链式；thenCombine/thenCompose 组合；allOf/anyOf 并行编排；exceptionally/handle 异常处理。必须传自定义线程池。
