单例模式 (Singleton)

创建型

定义：一个类在整个 JVM 中只有一个实例，并提供全局访问点。

// Spring 的"单例"和 GOF 的单例不同：
// GOF 单例：一个 ClassLoader 下只有一个实例
// Spring 单例：一个 IoC 容器中，同一个 beanName 只有一个实例
Spring 单例 Bean 的创建流程： getBean("userService") │ ▼ DefaultSingletonBeanRegistry 中的三级缓存： ┌────────────────────────────────────────────┐ │ ① singletonObjects (一级缓存, 成品池) │ ← 完全初始化的 Bean │ ② earlySingletonObjects(二级缓存, 半成品池) │ ← 已实例化, 未注入属性的 Bean │ ③ singletonFactories (三级缓存, 工厂池) │ ← 可生成半成品的 ObjectFactory └────────────────────────────────────────────┘ (解决循环依赖的核心机制)
Spring 中的应用：
· 默认作用域 @Scope("singleton") 就是单例
· DefaultSingletonBeanRegistry 用 ConcurrentHashMap 存储单例 Bean
· 三级缓存解决构造器注入的循环依赖问题 
// 源码位置：DefaultSingletonBeanRegistry
public class DefaultSingletonBeanRegistry {
    // 一级缓存：存放完全初始化好的单例Bean
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

    protected Object getSingleton(String beanName) {
        Object singletonObject = this.singletonObjects.get(beanName);
        if (singletonObject == null) {
            // 从三级缓存查找
        }
        return singletonObject;
    }
}

工厂模式 (Factory)

创建型

定义：不通过 new 创建对象，而是通过工厂方法来创建，将对象的创建和使用解耦。

Spring 中的应用（两种工厂模式并存）：
· 简单工厂 + 工厂方法：BeanFactory 和 ApplicationContext 是最典型的工厂
· FactoryBean 接口：当 Bean 的创建逻辑比较复杂时（如 MyBatis 的 Mapper 代理对象），实现这个接口 
// ① BeanFactory —— Spring 中最核心的工厂
public interface BeanFactory {
    Object getBean(String name);
    <T> T getBean(Class<T> requiredType);
}

// 调用方：
UserService service = context.getBean(UserService.class);
// 不需要知道 UserService 的实现类、构造参数、依赖关系 → 解耦

// ② FactoryBean —— 用来创建复杂对象
public interface FactoryBean<T> {
    T getObject();           // 返回实际的对象
    Class<?> getObjectType(); // 返回对象的类型
    default boolean isSingleton() { return true; }
}

// 示例：MyBatis 的 MapperFactoryBean
public class MapperFactoryBean<T> implements FactoryBean<T> {
    public T getObject() {
        return sqlSession.getMapper(mapperInterface); // 动态代理生成 Mapper
    }
}

// 注意：getBean("mapperFactoryBean") → 返回的是 getObject() 的结果（Mapper 代理）
//       getBean("&mapperFactoryBean") → 返回 FactoryBean 本身

代理模式 (Proxy)

结构型

定义：为其他对象提供一种代理，以控制对原对象的访问。AOP 的本质就是代理模式。

JDK 动态代理 vs CGLIB 代理

JDK 动态代理

基于接口。目标类必须实现接口。通过 Proxy.newProxyInstance() 和 InvocationHandler 创建代理。

Spring 默认策略：有接口 → JDK 代理

CGLIB 代理

基于继承。通过继承目标类并重写方法来创建代理。不能代理 final 类和方法。

Spring 默认策略：无接口 → CGLIB

AOP 代理调用链： 调用者 │ ▼ ┌──────────┐ │ 代理对象 │ (JDK 动态代理 / CGLIB 代理) │ (Proxy) │ └────┬─────┘ │ ▼ ┌──────────────────┐ │ @Before 通知 │ │ @Around 通知 │ ← 多个 Advisor 组成责任链 │ @AfterReturning │ │ @AfterThrowing │ │ @After 通知 │ └────────┬─────────┘ ▼ ┌──────────────────┐ │ 目标对象 │ (被代理的真实 Bean) │ (Target) │ └──────────────────┘ 
// Spring 选择代理方式的源码（DefaultAopProxyFactory）：
public AopProxy createAopProxy(AdvisedSupport config) {
    if (config.isOptimize() || config.isProxyTargetClass()
        || hasNoUserSuppliedProxyInterfaces(config)) {
        return new ObjenesisCglibAopProxy(config);  // → CGLIB
    }
    return new JdkDynamicAopProxy(config);       // → JDK 动态代理
}

// Spring Boot 2.x 默认使用 CGLIB（proxy-target-class: true）
AOP 失效的两个经典场景：
1. 同类方法调用：this.methodB() — this 是原始对象，不是代理，不走 AOP
2. private 方法：代理类无法重写 private 方法，AOP 失效

模板方法模式 (Template Method)

行为型

定义：在父类中定义算法的骨架（固定步骤），将某些步骤延迟到子类实现。Spring 中大量使用这种模式来"封装样板代码"。

Spring 中的应用（无处不在）：
· JdbcTemplate —— 获取连接/执行/释放连接交给模板，SQL + 结果映射交给你
· RestTemplate —— HTTP 请求的创建/发送/响应处理，模板搞定
· RedisTemplate / KafkaTemplate —— 同样的思路
· AbstractApplicationContext.refresh() —— 容器刷新的 12 步骨架 
// JdbcTemplate 的核心 —— query() 方法的简化版：
public <T> T query(String sql, RowMapper<T> rowMapper) {
    // ① 获取连接（模板替你做了）
    Connection conn = dataSource.getConnection();
    try {
        // ② 创建 Statement（模板替你做了）
        PreparedStatement ps = conn.prepareStatement(sql);
        // ③ 执行查询（模板替你做了）
        ResultSet rs = ps.executeQuery();
        // ④ 行映射 —— 只有这一步交给你来实现！
        return rowMapper.mapRow(rs, 0);
    } finally {
        // ⑤ 释放资源（模板替你做了）
        conn.close();
    }
}

// 你的代码只需关注"怎么把一行数据变成对象"：
jdbcTemplate.query("SELECT * FROM users", (rs, rowNum) ->
    new User(rs.getLong("id"), rs.getString("name"))
);
模板方法的核心思想：把不变的流程（获取连接、执行、释放）和变化的逻辑（SQL、结果映射）分离。你只写变化的，框架帮你处理不变的。

观察者模式 (Observer)

行为型

定义：一对多依赖关系。当主题（Subject）状态改变时，所有依赖的观察者自动收到通知。这是事件驱动架构的基础。

Spring 事件机制： ① 发布事件 ② 广播到所有匹配的 Listener ┌──────────┐ ┌──────────────────────────────┐ │ Publisher│──────→│ ApplicationEventMulticaster │ │ (Service)│ │ (事件广播器) │ └──────────┘ │ 遍历所有 Listener │ │ 调用 onApplicationEvent() │ └─────┬──────┬──────┬───────────┘ ┌───┘ │ └───┐ ▼ ▼ ▼ ┌──────┐ ┌──────┐ ┌──────┐ │订单 │ │短信 │ │积分 │ │监听器 │ │监听器 │ │监听器 │ └──────┘ └──────┘ └──────┘ 
// ① 定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Order order;
    public OrderCreatedEvent(Object source, Order order) {
        super(source); this.order = order;
    }
}

// ② 发布事件 —— 通过 ApplicationEventPublisher
@Service
public class OrderService {
    private final ApplicationEventPublisher publisher;
    public void createOrder(Order order) {
        // ... 保存订单 ...
        publisher.publishEvent(new OrderCreatedEvent(this, order));
    }
}

// ③ 监听事件
@Component
public class SmsListener {
    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        smsService.send(event.getOrder().getPhone());
    }
}

// ★ 默认是同步的！慢 Listener 会阻塞主流程。异步用 @Async + @EnableAsync
坑：Spring 事件默认是同步发布的，所有 Listener 在同一个线程中串行执行。如果某个 Listener 很慢（比如发邮件），会拖慢主流程。解决方案：@EventListener + @Async。

策略模式 (Strategy)

行为型

定义：定义一系列算法，把它们封装起来，让它们可以互相替换。策略模式让算法的变化独立于使用算法的客户端。

Spring 中的应用：
· ResourceLoader —— 根据不同前缀（classpath:/file:/http:）选择不同的资源加载策略
· HandlerMethodArgumentResolver —— 根据 Controller 方法参数注解（@RequestParam/@RequestBody...）选择不同的解析策略
· AccessDecisionManager —— Spring Security 中，根据投票策略决定是否放行
· Spring 的"策略注入" —— 同一接口的多个实现注入到 List 或 Map 中 
// ★ Spring 最优雅的策略模式用法：注入 List<接口> 或 Map<String, 接口>

// ① 定义策略接口
public interface PaymentStrategy {
    void pay(Order order);
    String getType();  // 返回 "alipay" / "wechat" / "unionpay"
}

// ② 实现多个策略
@Component("alipay")
public class AlipayStrategy implements PaymentStrategy {
    public void pay(Order order) { /* 支付宝支付 */ }
    public String getType() { return "alipay"; }
}

// ③ 使用时自动注入所有策略 → 无需 if-else！
@Service
public class PaymentService {
    // Spring 会自动把所有 PaymentStrategy 的实现注入到这个 Map 中
    // key = Bean 名称（"alipay", "wechat"）, value = 策略实例
    private final Map<String, PaymentStrategy> strategyMap;

    public PaymentService(Map<String, PaymentStrategy> strategyMap) {
        this.strategyMap = strategyMap;
    }

    public void pay(Order order, String payType) {
        PaymentStrategy strategy = strategyMap.get(payType);
        if (strategy != null) strategy.pay(order);
    }
}

// 新增支付方式只需加一个 @Component，不用改任何现有代码！→ 开闭原则

装饰器模式 (Decorator)

结构型

定义：在不改变原类的情况下，动态地给对象添加额外职责。比继承更灵活。

Spring 中的应用：
· BeanPostProcessor 的链式处理 —— 多个 BPP 依次"装饰"同一个 Bean
· ServerHttpRequestDecorator —— Spring Cloud Gateway 中，装饰请求对象以支持 Body 重复读取
· Spring Session 的 SessionRepositoryRequestWrapper 
// BeanPostProcessor 就是装饰器模式：
// 多个 BPP 排成一排，Bean 依次经过每个 BPP，每个 BPP "装饰"一下

// 例：Spring 中的 AutowiredAnnotationBeanPostProcessor
public class AutowiredAnnotationBeanPostProcessor implements BeanPostProcessor {
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        // 扫描 @Autowired @Value 注解 → 注入依赖
        return bean;  // 可能返回被装饰过的 Bean 或原 Bean
    }
}

// Spring Cloud Gateway 中的请求体缓存装饰器：
public class CachedBodyServerHttpRequest
    extends ServerHttpRequestDecorator {  // 装饰器抽象类
    // 缓存 RequestBody，让它可以多次读取
}
装饰器 vs 代理模式的区别：
代理模式关注控制访问（能不能调），装饰器模式关注增强功能（调用前后加什么）。但在实际代码中，两者经常混用。

责任链模式 (Chain of Responsibility)

行为型

定义：多个处理者形成一条链，请求沿链传递，每个处理者决定"自己处理"还是"交给下一个"。

Spring MVC 请求处理链： 请求 → FilterChain → DispatcherServlet │ ├── HandlerInterceptor.preHandle() │ ├── HandlerAdapter.handle() │ └── Controller.method() │ ├── HandlerInterceptor.postHandle() │ └── HandlerInterceptor.afterCompletion()
Spring 中的应用：
· Filter Chain：Servlet Filter 按顺序组成链，每个 Filter 调用 chain.doFilter() 传给下一个
· HandlerInterceptor：多个拦截器按注册顺序执行，preHandle 正向、postHandle/afterCompletion 反向
· Spring Security Filter Chain：十几层的安全过滤器链（认证、授权、CSRF、Session 管理等） 
// Spring Security 的过滤器链（简化版）：
// ChannelProcessingFilter
//   → SecurityContextPersistenceFilter
//   → UsernamePasswordAuthenticationFilter  ← 处理登录请求
//   → BasicAuthenticationFilter
//   → ExceptionTranslationFilter
//   → FilterSecurityInterceptor  ← 最后一道：授权检查

// 每个 Filter 可以选择：
//   ① 自己处理完 → 不往下传（如登录成功后返回 token）
//   ② 处理一部分 → chain.doFilter() 传给下一个
//   ③ 不处理 → chain.doFilter() 直接传

适配器模式 (Adapter)

结构型

定义：把一个类的接口转换成客户端期望的另一个接口，让原本不兼容的接口可以一起工作。

Spring 中的应用：
· HandlerAdapter：DispatcherServlet 用一种统一的方式调用不同类型的 Controller（@RequestMapping、HttpRequestHandler、Servlet 等）
· AdvisorAdapter：将不同类型的 Advice（MethodBeforeAdvice、AfterReturningAdvice）适配为统一的 MethodInterceptor 
// DispatcherServlet 中的核心逻辑：
protected HandlerAdapter getHandlerAdapter(Object handler) {
    for (HandlerAdapter adapter : this.handlerAdapters) {
        if (adapter.supports(handler)) {     // 这个 Adapter 能处理这个 handler 吗？
            return adapter;
        }
    }
}

// DispatcherServlet 有多个 HandlerAdapter：
//   RequestMappingHandlerAdapter  → 处理 @RequestMapping 注解的 Controller
//   HttpRequestHandlerAdapter     → 处理 HttpRequestHandler 接口的 Controller
//   SimpleControllerHandlerAdapter→ 处理旧式 Controller 接口

// DispatcherServlet 的调用：
mv = adapter.handle(request, response, handler);  // 统一的调用方式！
适配器模式的作用：让 DispatcherServlet 不必知道 Controller 的具体类型，只通过 adapter.handle() 统一调用。每种 Controller 类型有自己的适配器。新增 Controller 类型只需加一个适配器，不用改 DispatcherServlet。

建造者模式 (Builder)

创建型

定义：将一个复杂对象的构建与其表示分离，使得同样的构建过程可以创建不同的表示。

Spring 中的应用：
· BeanDefinitionBuilder —— 编程式构建 Bean 定义
· RequestMappingInfo.Builder —— 构建请求映射信息
· Spring Boot 的 SpringApplicationBuilder —— 构建 SpringApplication
· MockMvcRequestBuilders —— 测试中构建 Mock 请求 
// Spring 中建造者模式的几种体现

// ① BeanDefinitionBuilder
BeanDefinitionBuilder builder = BeanDefinitionBuilder
    .genericBeanDefinition(UserService.class)
    .addPropertyValue("timeout", 3000)
    .setLazyInit(true)
    .setScope("singleton");
BeanDefinition bd = builder.getBeanDefinition();

// ② SpringApplicationBuilder（Spring Boot）
new SpringApplicationBuilder()
    .sources(Application.class)
    .bannerMode(Banner.Mode.OFF)
    .profiles("dev")
    .run(args);

// ③ MockMvcRequestBuilders
MockHttpServletRequestBuilder request = MockMvcRequestBuilders
    .get("/api/users/{id}", 1)
    .contentType(MediaType.APPLICATION_JSON)
    .header("Authorization", "Bearer xxx");
建造者模式 vs 工厂模式：
工厂模式：一步创建对象，关注"创建什么"。
建造者模式：分步构建对象，关注"怎么一步一步搭起来"。参数多、配置复杂时用建造者。

设计模式在 Spring 中的分布总览

模式类型Spring 中的体现一句话理解
单例创建型Bean 默认 scope，三级缓存一个容器一个 Bean
工厂创建型BeanFactory, FactoryBean不 new，找工厂要
建造者创建型BeanDefinitionBuilder, MockMvcRequestBuilders一步一步搭对象
代理结构型AOP（JDK/CGLIB 动态代理）方法前后"搞事情"
装饰器结构型BeanPostProcessor, ServerHttpRequestDecorator一层一层包，逐步增强
适配器结构型HandlerAdapter, AdvisorAdapter不兼容 → 对个口子
模板方法行为型JdbcTemplate, RestTemplate, refresh()骨架固定，细节交给你
观察者行为型ApplicationEvent, @EventListener有事了通知你
策略行为型ResourceLoader, Map 注入多实现多种算法，自由切换
责任链行为型Filter Chain, Interceptor排队处理，一个接一个

十一、面试速记

Spring 中用了哪些设计模式？举 5 个例子。
单例：Bean 默认 scope=singleton，三级缓存。工厂：BeanFactory + FactoryBean。代理：AOP（JDK/CGLIB 动态代理）。模板方法：JdbcTemplate、RestTemplate。观察者：ApplicationEvent + @EventListener。策略：Controller 参数解析、ResourceLoader。责任链：Filter Chain、Interceptor。适配器：HandlerAdapter。
BeanFactory 和 FactoryBean 有什么区别？
BeanFactory 是 Spring 的 IoC 容器顶层接口，负责创建和管理所有 Bean。FactoryBean 是一个特殊的 Bean，它的 getObject() 方法返回的对象才是最终注册到容器中的 Bean。MyBatis 的 Mapper 代理对象就是通过 FactoryBean 创建的。
AOP 用的是什么设计模式？JDK 代理和 CGLIB 怎么选？
代理模式。目标类有接口 → JDK 动态代理（基于接口）；无接口 → CGLIB（基于继承）。Spring Boot 2.x 默认使用 CGLIB（proxy-target-class: true）。CGLIB 不能代理 final 类和方法。
JdbcTemplate 体现了什么设计模式？好处是什么？
模板方法模式。把"获取连接→执行→释放资源"这套固定流程封装起来，只把"SQL + 结果映射"通过回调（RowMapper）留给调用者。好处：减少样板代码，避免忘记关闭连接导致资源泄漏。
Spring 的事件机制是什么设计模式？默认同步还是异步？
观察者模式。默认是同步的（同一线程串行执行所有 Listener），如果某个 Listener 很慢会阻塞主流程。异步需要 @EventListener + @Async + @EnableAsync。
Spring 中如何优雅地实现策略模式？
定义策略接口 → 多个 @Component 实现 → 通过构造器注入 Map<String, 接口>（Spring 自动注入所有实现，key=beanName, value=实例）。新增策略只需加一个 @Component，符合开闭原则。
DispatcherServlet 中用到了什么设计模式？
HandlerAdapter 是适配器模式：把不同类型的 Controller（@RequestMapping、HttpRequestHandler、Servlet）适配成统一调用。HandlerMapping 是策略模式：根据请求找到对应的处理器。请求处理链是责任链模式。
Spring 的单例 Bean 是线程安全的吗？
不是。Spring 只保证单例 Bean 在容器中只有一个实例，但不保证线程安全。如果单例 Bean 有可变状态（如实例字段），多线程并发访问会出现问题。解决方案：无状态设计（推荐）、ThreadLocal、改用 prototype 作用域。
BeanPostProcessor 是什么设计模式？
装饰器模式。容器中有多个 BeanPostProcessor，Bean 在初始化前后会依次经过每个 BPP 的处理（postProcessBeforeInitialization → init → postProcessAfterInitialization）。每个 BPP 可以"装饰"Bean 或返回原 Bean。
Spring 中用到了哪些创建型设计模式？各自在什么场景？
单例：默认 Bean 作用域。工厂：BeanFactory / FactoryBean，解耦对象的创建和使用。建造者：BeanDefinitionBuilder、SpringApplicationBuilder，用于构建复杂配置对象。（没有原型模式，因为 Spring 的 prototype scope 只是每次 getBean 时创建新实例，不是通过 clone。）
