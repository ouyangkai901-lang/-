一、为什么需要消息队列？

消息队列解决的是系统之间通信的问题。在没有消息队列之前，系统间通信靠 HTTP/RPC 同步调用：

同步调用的问题

用户下单 → 扣库存 → 发短信 → 发积分
每个步骤串行等待，发短信慢 → 整个下单慢
发短信失败 → 下单失败（耦合）
流量高峰 → 后端撑不住（无缓冲）
每天只卖100个也要时刻准备10000个的并发能力

引入消息队列后

用户下单 → 扣库存 → 发消息到 Topic
↓
Topic 被 3 个消费者订阅：
· 短信服务消费 → 发短信
· 积分服务消费 → 发积分
· 数据分析消费 → 记日志
异步、解耦、削峰填谷

二、Kafka 核心架构

Kafka 逻辑架构： Producer ──→ ┌────────────────── Topic ────────────────────┐ ←── Consumer │ │ │ Partition-0: [msg0][msg1][msg2][msg3]... │ Producer ──→│ Partition-1: [msg0][msg1][msg2]... │←── Consumer Group │ Partition-2: [msg0][msg1][msg2][msg3][msg4] │ │ │ └───────────────────────────────────────────────┘ ↓ ↑ Broker-0 (Leader) Broker-1 (Follower) Broker-2 (Follower) (ISR 副本同步)
概念说明类比
BrokerKafka 实例节点服务器
Topic消息的逻辑分类数据库的表
PartitionTopic 的物理分片，有序队列分表
Producer生产者，发消息insert 语句
Consumer消费者，拉消息（pull 模式）select 语句
Consumer Group消费者组，组内负载均衡—
Offset消息在 Partition 中的位置自增 ID
ISRIn-Sync Replicas，同步中的副本MySQL 半同步从库
关键设计：Consumer Group 的负载均衡
同一个 Consumer Group 中的多个消费者，每个消费者独享一个或多个 Partition。
规则：一个 Partition 只能被同组的一个 Consumer 消费（保证顺序），但一个 Consumer 可以消费多个 Partition。
→ Partition 数量 ≥ Consumer 数量，否则会有 Consumer 空转。

三、Kafka 为什么这么快？

顺序写磁盘：Kafka 将消息追加写入 Partition 文件末尾，是顺序 I/O。机械硬盘顺序写 600MB/s，随机写 100KB/s。你以为是磁盘慢，其实是随机读写慢。
Page Cache（页缓存）：写入先写到 OS 的 Page Cache（内存），由 OS 决定何时刷盘。读也是优先从 Page Cache 读（热数据在内存）。
零拷贝（Zero Copy）：Consumer 拉数据时，数据从磁盘 → Page Cache → 网卡，不经过用户态内存拷贝。传统方式要拷贝 4 次，零拷贝只需 2 次。
批量 + 压缩：Producer 攒一批消息一起发（batch.size + linger.ms），Consumer 一次拉一批。配合 Snappy/LZ4 压缩，网络和磁盘 IO 大幅减少。
稀疏索引：Kafka 不为每条消息建索引，而是每隔一定字节建一个稀疏索引，找消息时先二分定位到最近的索引，再顺序扫。
一句话总结 Kafka 快的原因：顺序 I/O + 零拷贝 + Page Cache 缓存 + 批量压缩。不是在算法层面快，而是在操作系统层面让磁盘和网卡的物理极限被充分利用。

四、Kafka vs RocketMQ

维度KafkaRocketMQ
出身LinkedIn → Apache阿里 → Apache
吞吐量百万级 TPS（极高）十万级 TPS（高）
延迟ms 级ms 级（略低）
消息回溯按 Offset 回溯按时间回溯（更灵活）
事务消息支持（0.11+）原生支持，更成熟
顺序消息单 Partition 内有序全局顺序 + 分区顺序
延时消息不支持（需自己实现）18 个延时级别
消息过滤客户端过滤Broker 端 SQL 过滤 + Tag
死信队列需自己实现内置死信队列
运维依赖无（KRaft 去 ZooKeeper）NameServer（轻量级）
适用场景大数据流处理、日志收集在线业务、电商交易
选择建议：大数据/日志/流处理 → Kafka；电商交易/分布式事务/延时消息 → RocketMQ。国内互联网公司在线业务大多用 RocketMQ。

六、Kafka vs Pulsar

Pulsar 是新一代消息系统，核心卖点是计算存储分离（BookKeeper 存储 + Broker 无状态）。

维度KafkaPulsar
架构Broker 即存储，耦合计算存储分离（Broker 无状态 + BookKeeper）
多租户通过 Topic 前缀模拟原生 tenant/namespace 模型
消费模型Consumer Group4 种订阅模式（Exclusive/Shared/Failover/Key_Shared）
扩展性扩 Partition 需迁移数据Broker 无状态，秒级扩容
生态成熟度生态极丰富生态还在成长
适用场景大部分场景多租户平台、实时数仓

七、选型决策树

你的场景 → 选什么消息队列？ 你的场景是？ ├── 大数据流（日志、用户行为、埋点）→ Kafka ├── 电商交易（下单、支付、库存）→ RocketMQ │ └── 需要延时消息 / 事务消息 → 必选 RocketMQ ├── 复杂路由（Exchange、Topic 匹配规则）→ RabbitMQ ├── 多租户平台（SaaS，租户隔离）→ Pulsar ├── 运维简单、团队熟悉 → 选熟悉的不选新的 └── 数据量不大、不想维护集群 → Redis Stream / 直接 DB 轮询 你们团队（智能客服）： 用户行为埋点、对话日志 → Kafka（天然适合） 业务事件（订单、退款等） → 如果已有 RocketMQ 就用 RocketMQ，否则 Kafka 也够用

八、Kafka 核心操作

# ===== Topic 管理 =====
# 创建 Topic（3 分区，2 副本）
kafka-topics --bootstrap-server localhost:9092 \
  --create --topic order-events \
  --partitions 3 --replication-factor 2

# 查看 Topic 列表
kafka-topics --bootstrap-server localhost:9092 --list

# 查看 Topic 详情
kafka-topics --bootstrap-server localhost:9092 --describe --topic order-events

# 查看 Consumer Group 消费进度
kafka-consumer-groups --bootstrap-server localhost:9092 \
  --describe --group order-service

# ===== 生产/消费测试 =====
# 生产者（控制台）
kafka-console-producer --broker-list localhost:9092 --topic order-events

# 消费者（从头开始读）
kafka-console-consumer --bootstrap-server localhost:9092 \
  --topic order-events --from-beginning

Spring Boot 中集成 Kafka

// ===== application.yml =====
// spring.kafka.bootstrap-servers=localhost:9092
// spring.kafka.consumer.group-id=order-service
// spring.kafka.consumer.auto-offset-reset=earliest

// ===== Producer =====
@RestController
public class OrderController {
    private final KafkaTemplate<String, OrderEvent> kafka;

    @PostMapping("/orders")
    public void createOrder(@RequestBody Order order) {
        // 保存订单到 MySQL
        orderService.save(order);
        // 发消息通知下游
        kafka.send("order-events", order.getId(), new OrderEvent(order));
    }
}

// ===== Consumer =====
@Component
public class OrderEventConsumer {
    @KafkaListener(topics = "order-events")
    public void handle(OrderEvent event) {
        // 发短信、发积分...
        smsService.sendNotification(event.getOrder());
    }
}

九、常见问题与最佳实践

问题原因解决
消息丢失Producer 异步发、Broker 宕机、Consumer 自动提交 OffsetProducer: acks=all; Broker: min.insync.replicas=2; Consumer: 手动提交
消息重复Producer 重试、Consumer 处理完未提交 Offset 挂了Consumer 做幂等（唯一键去重/Redis setnx）

十、面试速记

Kafka 为什么这么快？
顺序写磁盘（600MB/s）+ Page Cache + 零拷贝（sendfile）+ 批量压缩 + 稀疏索引。充分利用了操作系统的底层能力。
Kafka 怎么保证消息不丢？
Producer: acks=all（等所有 ISR 确认）。Broker: replication.factor≥2, min.insync.replicas≥2。Consumer: 关闭自动提交，业务处理完再手动 commitSync。
Kafka 的 ISR 是什么？
In-Sync Replicas，与 Leader 保持同步的副本集合。Follower 落后太多（replica.lag.time.max.ms=30s）会被踢出 ISR。只有 ISR 中的副本才能选为 Leader。
Consumer Group 的 Rebalance 什么时候触发？
Consumer 数量变化（新增/下线）、Partition 数量变化、Consumer 超时未发心跳（max.poll.interval.ms）。Rebalance 期间所有 Consumer 停止消费！
Kafka vs RocketMQ 怎么选？
大数据流/日志 → Kafka。电商交易（事务消息、延时消息）→ RocketMQ。Kafka 吞吐高，RocketMQ 功能全（事务、延时、SQL过滤、死信）。
消息积压了怎么办？
紧急：新建 Topic 多加 Partition，写一个临时 Consumer 只做转发（不处理业务逻辑），快速清掉积压。长期：增加 Partition + 消费者实例、优化消费逻辑（批量、异步）。
