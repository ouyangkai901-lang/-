一、为什么需要搜索引擎？

假设你有一个电商系统，用户搜索"苹果手机 256G 红色"，MySQL 怎么做？

-- MySQL LIKE 做法：
SELECT * FROM products WHERE name LIKE '%苹果%' OR name LIKE '%手机%' OR name LIKE '%256G%';
-- 问题：
-- ① 无法分词："苹果手机"被当作一个整体，而不是"苹果"+"手机"
-- ② 无法相关性排序：哪个商品更匹配？不知道
-- ③ 全表扫描：百万级数据直接卡死
-- ④ 无法纠错：用户搜"平果手机"查不到任何东西
搜索引擎解决的核心问题：在海量数据中快速、精准、智能地找到用户想要的信息。核心能力：分词 → 倒排索引 → 相关性打分 → 高亮 → 聚合。

二、倒排索引 —— ES 的核心

ES 的快，根子在倒排索引（Inverted Index）。理解了它，就理解了搜索引擎的本质。

2.1 正排索引 vs 倒排索引

正排索引（MySQL B+Tree 索引）

文档ID → 文档内容

1 → "苹果手机很好用"
2 → "华为手机也不错"
3 → "苹果笔记本轻薄"

查"苹果" → 需要遍历所有文档，逐个检查内容

倒排索引（ES 的索引方式）

词条 → 包含该词条的文档列表

"苹果" → [1, 3]
"手机" → [1, 2]
"笔记本" → [3]

查"苹果" → 直接定位到文档 1 和 3，O(1) 查找

2.2 倒排索引的构建过程

原始文档 → 倒排索引： ① 文档 → ② 分词 → ③ 词条标准化 → ④ 建立倒排表 文档1: "The quick brown fox jumped over the lazy dog" 文档2: "The quick brown fox is quick" 分词 + 标准化后： ┌──────────┬──────────────┐ │ 词条 │ 文档列表 │ ├──────────┼──────────────┤ │ quick │ [1, 2] │ ← 文档2中 quick 出现了2次, 还需要存位置 │ brown │ [1, 2] │ │ fox │ [1, 2] │ │ jump │ [1] │ │ dog │ [1] │ └──────────┴──────────────┘ 实际存储结构更复杂： Term → (DocID, TF, [position1, position2, ...]) "quick" → (1, 1, [2]), (2, 2, [3, 7]) ↑ ↑ 词频TF 位置索引(支持短语查询)

2.3 为什么倒排索引这么快？

底层是 Term Dictionary（排序的词条列表），查找时做二分查找或基于 FST（有限状态转换器）的前缀匹配，不在磁盘上随机读写
Term Dictionary 有 Term Index 加速：Term Index 是一棵 Trie 树（前缀树），内存中就能定位到词条在 Term Dictionary 的哪个 Block
Posting List（文档列表）用 SkipList + Frame of Reference 压缩，跳跃读取 + 高效压缩，IO 极低
多条件查询时，Posting List 求交集（如"苹果 AND 手机"），用跳表或 BitMap 做集合运算，效率极高
一句话总结：MySQL 是"拿着文档找词"，ES 是"拿着词找文档"。当数据量大且查询条件是"包含某个词"时，倒排索引效率碾压 B+Tree。

三、ES 核心概念与架构

ES 逻辑结构： Cluster (集群) ├── Node-1 (节点, 一个 ES 进程) │ ├── Shard-0 (P) ← 主分片 │ ├── Shard-1 (R) ← 副本分片 (来自 Node-2 的 Shard-1 副本) │ └── Shard-2 (R) │ ├── Node-2 │ ├── Shard-0 (R) │ ├── Shard-1 (P) │ └── Shard-2 (R) │ └── Node-3 ├── Shard-0 (R) ├── Shard-1 (R) └── Shard-2 (P) Index (索引, 类似 MySQL 的 Database) └── Type (类型, 7.x 开始废弃, 一个 Index 只有 _doc) └── Document (文档, 类似 MySQL 的一行) └── Field (字段, 类似 MySQL 的一列)
ES 概念MySQL 类比说明
IndexDatabase索引，一个业务一套索引
Type (7.x 废弃)TableES 7.x 后一个 Index 只能有一个 _doc Type
DocumentRowJSON 格式的一条数据
FieldColumn文档中的字段
MappingSchema定义字段的类型、分词器等
Shard分表数据分片，分布式存储
Replica从库副本，高可用 + 提升查询吞吐
ES 分布式写入流程（关键！）：
请求 → 任意节点（协调节点）→ 根据 doc_id 路由到主分片 → 主分片写入 + 同步到副本分片 → 副本分片全部成功后返回确认
shard = hash(routing) % number_of_primary_shards（routing 默认是 doc_id）

四、ES vs Solr —— 搜索引擎内战

维度ElasticsearchSolr
底层引擎LuceneLucene（和 ES 一样）
分布式天生分布式，集群配置简单依赖 ZooKeeper，搭建复杂
实时搜索近实时（1秒）稍慢
JSON 支持原生 JSON，RESTful APIXML/JSON 都支持
生态与社区ELK Stack 全生态，社区活跃相对小众
大数据量PB 级，水平扩展强TB 级够用，更大不如 ES
静态数据实时搜索富文本文档搜索更精确
使用门槛低（RESTful API）中（配置文件较多）
选 Solr 的唯一理由：你的数据是大量的静态文档（PDF、Word 等），且需要精确的全文搜索体验。大多数互联网场景无脑选 ES。

五、ES vs MongoDB —— 搜索 vs 存储

这是最常见的混淆——"MongoDB 也可以建索引，也能搜文本，为什么还要 ES？"

维度ElasticsearchMongoDB
定位搜索引擎（专为搜索而生）文档数据库（NoSQL 存储）
核心索引倒排索引B-Tree 索引
分词内置中文分词器（ik、jieba）不支持中文分词
相关性打分TF-IDF / BM25，排序准确不支持
聚合分析强大的 Aggregation（统计、分组）Aggregation Pipeline（功能较弱）
高亮原生支持搜索词高亮不支持
模糊查询拼写纠错、拼音搜索仅支持正则（慢）
事务不支持4.0+ 支持多文档事务
写入性能中等（近实时刷新）高（直接写内存）
数据可靠性写入有 translog，但不是 ACID可配置 Write Concern
最佳实践：不要把 ES 当数据库用！
MongoDB/MySQL 做主存储，数据同步到 ES 做搜索。常见同步方案：
· Canal（阿里开源）：监听 MySQL binlog → 同步到 ES
· Logstash：定时从数据库拉数据 → 写入 ES
· 应用层双写：写入 MySQL 同时写入 ES（注意事务一致性）

六、ES vs MySQL 全文索引

MySQL 5.6+ 也有 FULLTEXT 索引（InnoDB），为什么不用？

维度MySQL FULLTEXTElasticsearch
分词内置（空格分词），不支持中文ik 分词器 / jieba 分词
相关性排序简单，不可定制BM25 + 自定义打分
百万级性能慢（同时承担 OLTP 压力）专门优化，毫秒级
高亮/聚合不支持原生支持
与 OLTP 互相影响搜索 + 写入抢资源独立服务，不影响主库

七、选型决策树

你的场景 → 选什么？ 需要搜索？ ├── 数据量不大(<10万)，简单 LIKE 够用 → MySQL LIKE 就行，别过度设计 ├── 数据量几十万，需要模糊搜索 → MySQL FULLTEXT + ngram 分词器 ├── 需要中文分词 / 相关性排序 / 高亮 → Elasticsearch │ ├── 数据是静态文档(PDF/Word) → 也可考虑 Solr │ └── 一般业务搜索 → 无脑 ES ├── 只是查日志 → ES + Kibana (ELK) └── 只是简单的 NoSQL 存储 + 偶尔搜 → MongoDB，不要引入 ES 增加复杂度

八、ES 基础 CRUD 操作

# ===== 索引管理 =====
# 创建索引（设置分词器）
PUT /products
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "analysis": {
      "analyzer": { "ik_smart": { "type": "custom", "tokenizer": "ik_smart" } }
    }
  },
  "mappings": {
    "properties": {
      "title":    { "type": "text",    "analyzer": "ik_max_word" },
      "price":    { "type": "double" },
      "category": { "type": "keyword" },
      "created":  { "type": "date" }
    }
  }
}

# ===== 文档 CRUD =====
# 新增 / 全量替换（指定 ID）
PUT  /products/_doc/1  { "title": "苹果 iPhone 15 Pro 256G", "price": 8999, "category": "手机" }

# 新增（自动生成 ID）
POST /products/_doc     { "title": "华为 Mate 60 Pro", "price": 6999 }

# 查询
GET  /products/_doc/1

# 部分更新
POST /products/_update/1  { "doc": { "price": 8499 } }

# 删除
DELETE /products/_doc/1

# ===== 搜索 =====
# 全文搜索（会分词）
GET /products/_search
{
  "query": {
    "match": { "title": "苹果手机 256G" }
  }
}

# 精确匹配（不分词）
GET /products/_search
{ "query": { "term": { "category": "手机" } } }

# 范围查询 + 排序
GET /products/_search
{
  "query": {
    "bool": {
      "must":     [ { "match": { "title": "手机" } } ],
      "filter":   [ { "range": { "price": { "gte": 5000, "lte": 10000 } } } ]
    }
  },
  "sort": [ { "price": "asc" } ],
  "from": 0, "size": 20
}

# 聚合：按分类统计数量和平均价格
GET /products/_search
{
  "size": 0,
  "aggs": {
    "by_category": {
      "terms": { "field": "category" },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } }
      }
    }
  }
}

十、面试速记

ES 的倒排索引是什么？为什么比 MySQL 搜得快？
倒排索引：词条 → 文档列表。搜索时直接 O(1) 定位到包含该词的文档。"拿着词找文档" vs MySQL "拿着文档找词"。底层用 FST + SkipList + Frame of Reference 压缩，内存高效、IO 极少。
ES 和 MongoDB 区别，什么时候用谁？
ES 是搜索引擎（倒排索引、分词、打分、高亮），MongoDB 是文档数据库（B-Tree、事务、高写入）。MySQL/MongoDB 主存储 + Canal 同步到 ES 搜索，是最佳实践。
ES 写入流程？为什么说是"近实时"？
协调节点路由到主分片 → 主分片写入 translog + 内存 buffer → 默认 1s refresh 刷新 segment 可搜索 → translog 定期 fsync 到磁盘。refresh 间隔（默认 1s）就是"近实时"的原因。
ES 怎么保证数据不丢？
translog（预写日志），每次写入先记录 translog。节点崩溃后从 translog 恢复。translog 默认每 5s 或每次请求后 fsync（可配置）。副本分片也提供冗余。
ES 分片数怎么设置？
主分片数创建后不可改（影响路由 hash）。建议：单节点数据量不超过 50GB/分片，总分片数 = 数据量/50GB。小数据量用 1-3 个分片即可。
