三、重点语法 · 代码对比

3.1 Record 类 —— 告别写 getter/setter/equals/hashCode

JDK 8 写法（一个简单的 POJO）
public class User {
    private final String name;
    private final int age;

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }
    public String getName() { return name; }
    public int getAge()  { return age; }

    // equals / hashCode / toString
    // 至少要 30+ 行代码...
}
JDK 17+ 一行搞定
public record User(String name, int age) {}

// 自动生成：
//   构造器、getter、equals
//   hashCode、toString
// 自动不可变（final）

3.2 模式匹配 —— 告别冗长的 instanceof + 强制转型

JDK 8 写法
if (obj instanceof String) {
    String s = (String) obj;     // 强制转型
    System.out.println(s.length());
}
JDK 17+ 模式匹配
if (obj instanceof String s) {  // 直接绑定变量！
    System.out.println(s.length());
}

3.3 Switch 表达式 + 模式匹配 —— 从"语句"到"表达式"

JDK 8 冗长的 if-else 链
String result;
if (obj instanceof Integer i) {
    result = "int: " + i;
} else if (obj instanceof String s) {
    result = "string: " + s;
} else {
    result = "unknown";
}
JDK 21 switch 模式匹配
String result = switch (obj) {
    case Integer i  -> "int: " + i;
    case String s   -> "string: " + s;
    case null        -> "null";
    default          -> "unknown";
};

3.4 密封类 —— 限制继承的"可控多态"

JDK 8 —— 无法限制子类
public abstract class Shape {
    // 任何人/任何模块都可以继承
    // 无法穷举子类型
}
class Circle  extends Shape { }
class Square  extends Shape { }
class Unknown extends Shape { } // 不受控
JDK 17+ 密封类
public sealed class Shape
    permits Circle, Square { }

final class Circle extends Shape { }
final class Square extends Shape { }

// 配合 switch 模式匹配
// 编译器知道你穷举了所有子类型
// 不需要写 default 分支！

四、垃圾收集器演进

JDK 8

SerialParallelCMSG1
· 默认 GC：Parallel（吞吐量优先）
· G1 可用但不够成熟
· CMS 已标记 @Deprecated
· 无 ZGC、无 Shenandoah
· PermGen → Metaspace

JDK 17

SerialG1ParallelZGCShenandoah
· 默认 GC：G1（均衡延迟与吞吐）
· CMS 彻底删除
· ZGC 可用（低延迟 <1ms）
· Shenandoah 可用
· G1 NUMA 感知优化

JDK 21

SerialG1Parallel分代ZGCShenandoah
· 默认 GC：G1
· 分代 ZGC：正式推出
└ 年轻代+老年代分离
└ 吞吐提升 10%+
└ 暂停 <1ms（Heap ≤ 16TB）

五、JDK 21 杀手锏：虚拟线程（Virtual Threads）

5.1 传统线程 vs 虚拟线程

维度平台线程（Thread）虚拟线程（Virtual Thread）
本质 1:1 映射到 OS 线程 JVM 管理的轻量级"协程"
创建成本 ~1ms + ~1MB 栈 ~1μs + 几 KB 栈（可动态伸缩）
上下文切换 用户态 ↔ 内核态 纯用户态切换（无内核参与）
最大数量 约 1000-5000（受限于 OS） 可达百万级
阻塞代价 线程阻塞 → OS 挂起 阻塞时释放底层平台线程，不浪费

🔧 虚拟线程运行模型

大量虚拟线程复用少量平台线程（CPU 核数），阻塞时不占用平台线程
虚拟线程 1 虚拟线程 2 虚拟线程 3 虚拟线程 4 虚拟线程 5 ... x 10万
▼ 调度到少量平台线程 ▼
平台线程 A 平台线程 B 平台线程 C 平台线程 D

5.2 代码对比：1 万并发请求

JDK 8 —— 线程池方式
// 最多开 200 个线程 (CPU 已打满)
var pool = Executors
    .newFixedThreadPool(200);

for (int i = 0; i < 10000; i++) {
    pool.submit(() -> {
        callRemoteAPI();    // IO 阻塞
    });
}

// 问题：200 个线程只有 200 个并发
// 其余 9800 个任务排队等待
JDK 21 —— 虚拟线程方式
// 每个请求一个虚拟线程
try (var executor = Executors
        .newVirtualThreadPerTaskExecutor()) {

    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> {
            callRemoteAPI();  // 阻塞时释放平台线程
        });
    }
}

// 1 万个虚拟线程同时运行！
// IO 阻塞时自动释放底层平台线程
// 不需要线程池参数调优
虚拟线程最佳场景：IO 密集型任务（HTTP 请求、数据库查询、消息消费）。它们让"一个请求一个线程"变成现实，线程池调优的痛苦一去不返。

不适用场景：CPU 密集型任务（计算、加密、压缩）。这些场景下虚拟线程的协作式调度反而可能降低吞吐。

六、JDK 21 结构化并发（预览）

结构化并发的核心思想：父子任务的边界清晰、生命周期可控、异常传播有迹可循。

JDK 8 —— CompletableFuture
public Response handle() throws Exception {
    var f1 = CompletableFuture
        .supplyAsync(this::queryUser);
    var f2 = CompletableFuture
        .supplyAsync(this::queryOrder);

    // f1 失败不会自动取消 f2
    // f2 还在浪费资源跑
    return f1.thenCombine(f2, ...).get();
}
JDK 21 —— 结构化并发
public Response handle() throws Exception {
    try (var scope = new StructuredTaskScope
            .ShutdownOnFailure()) {

        var f1 = scope.fork(this::queryUser);
        var f2 = scope.fork(this::queryOrder);

        scope.join();
        scope.throwIfFailed();
        // f1 失败 → f2 自动取消！
        return combine(f1.get(), f2.get());
    } // try-with-resource 自动等待所有子任务
}

九、从 JDK 8 迁移注意事项

9.1 常见兼容性问题

问题原因解决方案
sun.misc.Unsafe 不可用 JDK 内部 API 被强封装 加 JVM 参数 --add-opens java.base/sun.misc=ALL-UNNAMED（短期）；升级依赖（长期）
反射访问 java.lang 私有字段 模块化系统限制 加 --add-opens java.base/java.lang=ALL-UNNAMED
Lombok 不兼容 旧版本不支持 JDK 17+ 升级 Lombok ≥ 1.18.22
Groovy / Kotlin 等 可能版本太旧 升级到最新版本
GC 日志格式变化 JDK 9+ 统一为 -Xlog:gc 替换旧的 -XX:+PrintGC
JAXB / javax.annotation 等移除 Java EE 模块被移除 手动加 Maven 依赖
迁移检查清单：
1. Maven/Gradle 编译 JDK 版本改为 17/21
2. 检查所有依赖的兼容性（Lombok / Netty / ASM 等底层库最易出问题）
3. 在测试环境验证 GC 表现（尤其从 Parallel → G1）
4. 检查 --add-opens 参数是否可以减少（逐步修复）

十、生产环境选型决策指南

继续用 JDK 8
如果你符合以下情况
维护老项目 中间件不兼容 依赖无法升级 短期不重构 Oracle 要付费
升级到 JDK 17
如果你符合以下情况
Spring Boot 3.x 新项目启动 追求代码简洁 安全合规要求 免费 LTS 支持
升级到 JDK 21
如果你符合以下情况
高并发 IO 场景 微服务网关 消息消费应用 追求极低延迟 分代 ZGC 需求
建议：
· 老项目维护 → JDK 8 还够用，但尽早规划升级路径
· 新项目启动 → 直接上 JDK 17（Spring Boot 3.x 的最低要求）
· 高并发/IO 密集 → JDK 21（虚拟线程 + 分代 ZGC 是巨大收益）
· 中间件选型时优先考虑 JDK 17/21 兼容性，避免技术债务

JDK 8 稳定可靠但已显老态 · JDK 17 是新项目的保守选择 · JDK 21 是面向未来的激进选择
