一、为什么需要向量数据库？

传统搜索（ES / MySQL）解决的是精确匹配和关键词匹配的问题。但在智能客服场景中，用户问：

用户输入："我的快递怎么还没到？"
FAQ 库里有："物流超时如何查询？"

关键词匹配：0 个共同词 → 搜不到 ❌
语义匹配：  两句话意思一样 → 应该匹配 ✅

向量数据库就是用来做语义搜索的：把文本转成向量，通过计算向量之间的"距离"，找到语义最相似的内容。

向量数据库在智能客服中的位置： 用户问题："我的快递怎么还没到？" │ ▼ Embedding 模型 (text-embedding / BGE 等) │ ▼ [0.023, -0.415, 0.891, ...] ← 768 维向量 │ ▼ Qdrant 向量检索 → 找最相似的 Top-K 个 FAQ │ ▼ 返回: "物流超时如何查询？"(相似度 0.95) "订单物流进度怎么查？"(相似度 0.87) "包裹丢失怎么办？"(相似度 0.72)

二、Embedding —— 把文本/图片变成向量

Embedding 是整个 RAG（检索增强生成）流程的第一步。简单说，就是用一个训练好的模型，把一段文本映射成一个固定长度的浮点数数组（向量）。

模型维度中文效果适用场景
OpenAI text-embedding-ada-0021536一般通用（需调 API）
OpenAI text-embedding-3-large3072较好通用（需调 API）
BGE-large-zh1024优秀中文语义检索首选
m3e-base768优秀中文开源（轻量）
text2vec-large-chinese1024良好中文
 
// 用 Spring AI 或 LangChain4j 调用 Embedding 模型
// 伪代码：
String userQuestion = "我的快递怎么还没到？";
float[] vector = embeddingModel.embed(userQuestion);  // → float[768]

// 在 Qdrant 中搜索最相似的 5 条 FAQ
List<ScoredPoint> results = qdrantClient.search(
    new SearchRequest()
        .setCollectionName("faq")
        .setQueryVector(vector)
        .setLimit(5)
);

三、向量相似度检索原理

两个向量之间怎么算"相似"？有三种常见的距离度量方式：

度量方式公式特点Qdrant 默认
余弦相似度 Cosinecos = (A·B)/(|A|*|B|)关注方向，不关注长度。值越大越相似推荐
欧氏距离 Euclidean√Σ(Ai-Bi)²关注绝对距离。值越小越相似—
点积 Dot ProductA·B适合归一化后的向量—
现实中：向量数据库不会做"全量扫描计算相似度"（太慢），而是用 ANN（近似最近邻）算法，比如 HNSW（Hierarchical Navigable Small World）图算法，在毫秒级从百万向量中找到最相似的 Top-K。

四、Qdrant 核心架构

Qdrant 架构： ┌────────────┐ │ REST API │ ← gRPC / HTTP │ (Rust) │ └─────┬──────┘ │ ┌─────┴──────┐ │ Collection │ ← 类似数据库的 Table │ ├── config │ 向量维度、距离度量方式 │ └── points │ 每个 point = 向量 + payload(元数据) └─────┬──────┘ │ ┌─────┴──────┐ │ Segments │ ← 数据分段存储 │ ├── 不可变 Segment (存盘) │ └── 可变 Segment (内存, 定期刷盘) └─────┬──────┘ │ ┌─────┴──────┐ │ HNSW 索引 │ ← 导航小世界图，快速近似搜索 └────────────┘ 核心特点： • Rust 编写，单机性能极强 • 支持 Payload 过滤（向量搜索 + 结构化条件过滤同时做） • 支持稠密向量(Dense) + 稀疏向量(Sparse) 混合搜索 • 内置量化（Scalar/Product Quantization）减少内存占用

七、向量数据库 vs 传统数据库 + 向量扩展

很多传统数据库也开始支持向量检索了：PostgreSQL 的 pgvector、Elasticsearch 的 dense_vector、Redis 的 RediSearch。那到底要不要单独引入向量数据库？

方案优点缺点适用
pgvector (PostgreSQL)不需要新组件，事务一致百万级向量后性能下降明显小规模、需要强一致性
ES dense_vector关键词+向量混合搜索向量性能不如专用数据库已有 ES 且数据不大
专用向量数据库 (Qdrant/Milvus)性能最强、功能最全多一个组件要维护核心语义搜索场景
选择建议：如果你们的智能客服是核心业务、FAQ 数量十万级以上 → 专用向量数据库（Qdrant）。如果只是一个小功能、几千条 FAQ → pgvector 就够用，别为了用而用。

八、选型决策树

向量数据库选型： 你的场景？ ├── 数据量小（<10万条），已有 PostgreSQL → pgvector ├── 快速验证，不想维护服务 → Pinecone（贵，数据出国） ├── 私有化部署，百万～千万级 → Qdrant（部署简单，性能好） ├── 私有化部署，亿级以上 → Milvus（分布式成熟） ├── 需要 GraphQL / 内置 AI 模块 → Weaviate └── 智能客服场景（你们的场景）→ Qdrant 最合适 └── 理由：部署简单(docker run)、Rust 性能强、 支持 Payload 过滤（按分类/状态过滤 FAQ）

九、Qdrant 核心操作

# ===== 创建 Collection =====
PUT /collections/faq
{
  "vectors": {
    "size": 768,
    "distance": "Cosine"
  }
}

# ===== 插入向量点（带 Payload）=====
PUT /collections/faq/points
{
  "points": [
    {
      "id": 1,
      "vector": [0.023, -0.415, ...],   // 768 维向量
      "payload": {
        "question": "物流超时如何查询？",
        "answer": "请打开APP→我的→帮助中心→物流查询，输入运单号即可查询。",
        "category": "物流",
        "priority": 1
      }
    }
  ]
}

# ===== 向量相似度搜索（不带过滤）=====
POST /collections/faq/points/search
{
  "vector": [0.031, -0.389, ...],
  "limit": 5,
  "with_payload": true
}

# ===== 向量搜索 + Payload 过滤 =====
POST /collections/faq/points/search
{
  "vector": [...],
  "limit": 5,
  "filter": {
    "must": [
      { "key": "category", "match": { "value": "物流" } }
    ]
  }
}

Java 客户端操作

// Maven 依赖：io.qdrant:client:1.7+
QdrantClient client = new QdrantClient(
    QdrantGrpcClient.newBuilder("localhost", 6334, false).build());

// 搜索
List<ScoredPoint> results = client.searchAsync(
    SearchPoints.newBuilder()
        .setCollectionName("faq")
        .addAllVector(queryVector)
        .setLimit(5)
        .setWithPayload(WithPayloadSelectorFactory.enable(true))
        .build()
).get();

// results 中包含每个结果的 id、score、payload
for (ScoredPoint point : results) {
    System.out.println("score: " + point.getScore());
    System.out.println("answer: " + point.getPayload().get("answer"));
}

Spring AI 集成（更简单）

// application.yml:
// spring.ai.qdrant.host=localhost
// spring.ai.qdrant.port=6334
// spring.ai.qdrant.collection-name=faq

@Autowired
private VectorStore qdrantVectorStore;

public List<Document> searchSimilarFAQ(String userQuestion) {
    return qdrantVectorStore.similaritySearch(
        SearchRequest.query(userQuestion).withTopK(5)
    );
}

十、面试速记

什么是向量数据库？和传统数据库有什么区别？
传统数据库做精确匹配（"年龄=25"），向量数据库做语义相似度匹配（"意思相近"）。核心：用 Embedding 模型将文本转成向量 → 用 ANN 算法（HNSW）快速找最相似的向量。解决的是"理解语义"而非"匹配关键词"的问题。
Qdrant 和 Milvus 怎么选？
百万～千万级 + 想要部署简单 → Qdrant（单二进制、Rust 性能强、Payload 过滤灵活）。亿级以上 + 需要强大分布式 → Milvus（CNCF 孵化、分布式成熟、但部署组件多）。
Embedding 模型怎么选？
中文语义检索首选 BGE-large-zh（1024维）或 m3e-base（768维，轻量）。追求效果用 OpenAI text-embedding-3-large（但需 API、数据出国）。维度越高效果越好但存储和检索成本也越高。
HNSW 算法是什么？
分层可导航小世界图。构建多层图结构，上层稀疏（长距离跳跃）、下层稠密（精确搜索）。检索时从顶层快速接近目标区域，逐层下降。时间复杂度 O(log N)，百万级向量毫秒级检索。
什么时候用 pgvector 就够了，不需要专门的向量数据库？
数据量小（<10万）、已有 PostgreSQL、向量搜索是辅助功能而非核心。反之，向量搜索是核心功能、数据量大（十万+）、要求低延迟 → 专用向量数据库。
