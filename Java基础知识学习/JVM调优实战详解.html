一、JVM 内存模型（运行时数据区）

JVM 启动时会从操作系统申请一大块内存，然后划分成若干区域。理解这些区域的职责是 JVM 调优的基础。

JVM 运行时数据区全景图： ┌──────────────────────────────────────────────────────┐ │ JVM 进程 │ │ │ │ ┌────────────────────┐ ┌────────────────────┐ │ │ │ 程序计数器 │ │ 本地方法栈 │ │ │ │ (每个线程一个) │ │ (每个线程一个) │ │ │ └────────────────────┘ └────────────────────┘ │ │ │ │ ┌────────────────────┐ ┌────────────────────┐ │ │ │ 虚拟机栈 │ │ 堆 (Heap) │ │ │ │ (每个线程一个) │ │ (所有线程共享) │ │ │ │ ┌─── 栈帧 ───┐ │ │ ┌── 新生代 ──┐ │ │ │ │ │ 局部变量表 │ │ │ │ Eden │ │ │ │ │ │ 操作数栈 │ │ │ │ S0 S1 │ │ │ │ │ │ 动态链接 │ │ │ └────────────┘ │ │ │ │ │ 返回地址 │ │ │ ┌── 老年代 ──┐ │ │ │ │ └───────────┘ │ │ │ Old Gen │ │ │ │ └────────────────────┘ │ └────────────┘ │ │ │ └────────────────────┘ │ │ ┌────────────────────────────────────────────┐ │ │ │ 元空间 (Metaspace) │ │ │ │ (所有线程共享, 直接内存) │ │ │ │ 存放: 类元数据 / 方法信息 / 常量池 │ │ │ └────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────┘

JDK 8 的重大变化：永久代 → 元空间
JDK 7 及之前：HotSpot 使用永久代 (PermGen) 存放类元数据，默认大小约 80MB，容易 OOM。
JDK 8 开始：永久代被移除，改为元空间 (Metaspace)，使用直接内存，默认无上限（受物理内存限制），通过 -XX:MaxMetaspaceSize 控制。

1.1 虚拟机栈 —— 一个方法一个栈帧

每调用一个方法，JVM 就创建一个栈帧压入栈顶；方法执行完毕，栈帧弹出。栈帧里面装着：

栈帧内部结构： ┌──────────────┐ │ 局部变量表 │ ← 方法参数 + 方法内局部变量（基本类型、对象引用） │ 操作数栈 │ ← 字节码指令执行时的临时计算区域 │ 动态链接 │ ← 指向运行时常量池中该方法的符号引用 │ 返回地址 │ ← 方法执行完后回到哪里 └──────────────┘ 例：public int add(int a, int b) { return a + b; } 局部变量表：this(隐含), a, b 操作数栈： iload a → iload b → iadd → ireturn

二、堆内存结构 —— 分代模型

绝大多数对象都在堆上分配。JVM 把堆分成 新生代 (Young Generation) 和 老年代 (Old Generation)，分别使用不同的 GC 策略。这就是"分代收集"理论的基础。

堆内存分代结构（JDK 8 默认比例）： ┌──────────────────────────────────────────────────────┐ │ Heap (堆) │ │ 默认: 新生代 1/3, 老年代 2/3 │ │ │ │ ┌────────────────────┐ ┌──────────────────────────┐│ │ │ 新生代 (Young) │ │ 老年代 (Old/Tenured) ││ │ │ │ │ ││ │ │ ┌──Eden──┐ 8/10 │ │ 存放长期存活的对象 ││ │ │ │ │ │ │ 或大对象(直接进老年代) ││ │ │ └────────┘ │ │ ││ │ │ ┌─S0─┐ ┌─S1─┐ │ │ GC: Major GC / Full GC ││ │ │ │ │ │ │1/10 │ │ 通常较慢, 应尽量避免 ││ │ │ └────┘ └────┘ │ │ ││ │ │ │ │ ││ │ │ GC: Minor GC │ │ ││ │ │ 频繁, 速度快 │ │ ││ │ └────────────────────┘ └──────────────────────────┘│ └──────────────────────────────────────────────────────┘

为什么 S0 和 S1 总有一个是空的？Minor GC 时，把 Eden + 有对象的 Survivor 中的存活对象，复制到另一个空的 Survivor。这就是"标记-复制"算法。

2.1 什么对象进老年代？

长期存活的对象：每熬过一次 Minor GC，年龄 +1。年龄超过 -XX:MaxTenuringThreshold（默认 15），晋升老年代
大对象：超过 -XX:PretenureSizeThreshold 的对象直接在老年代分配
Survivor 放不下：Minor GC 后 Survivor 空间不够，多余的对象直接进老年代（分配担保）
动态年龄判断：Survivor 中相同年龄的对象大小之和超过 Survivor 一半，该年龄及以上直接晋升

四、GC 算法 —— 三种基础算法

4.1 标记-清除 (Mark-Sweep)

过程：标记存活对象 → 清除未标记对象 标记前： [A][B][ ][C][ ][D] 标记后： [A][B][ ][C][ ][D] ← A/B/C/D 存活 ↑↑ ↑ ↑ ↑ 清除后： [A][B] [C] [D] ← 不连续，有碎片 优点：不需要移动对象 缺点：内存碎片化，大对象可能分配失败

4.2 标记-复制 (Mark-Copy) —— 新生代首选

过程：标记存活对象 → 复制到新区域 → 清空旧区域 标记前： [A][B][ ][C][ ][D] (from 区) ↑↑ ↑ ↑ ↑ 复制后： [A][B][C][D] (to 区, 紧凑排列) [ ][ ][ ][ ][ ][ ] (from 区, 清空) 优点：无碎片，分配只需指针碰撞，极快 缺点：浪费空间（一半内存闲置 → 所以新生代用 8:1:1 比例）

4.3 标记-整理 (Mark-Compact) —— 老年代使用

过程：标记存活对象 → 全部移到内存一端 → 清理边界外 标记前： [A][B][ ][C][ ][D] 整理后： [A][B][C][D]____________ (存活对象在一端, 边界外全清除) 优点：无碎片，不浪费空间 缺点：需要移动对象、更新引用，STW 时间长
新生代用"复制"，老年代用"整理/清除"的原因是：
新生代中绝大多数对象"朝生夕死"，每次 GC 活下来的很少 → 复制成本低。
老年代中对象存活率高，复制成本大 → 用标记-清除或标记-整理。

五、GC 收集器 —— 7 种收集器详解

GC 收集器是 GC 算法的具体实现。不同收集器适用于不同场景，关键是吞吐量 vs 延迟的权衡。

HotSpot 收集器全景（连线 = 可搭配使用）： ┌─────────────────────────────────────────────────────┐ │ 新生代收集器 │ │ Serial ────────┐ │ │ ParNew ────────┤ │ │ Parallel │ │ │ Scavenge ──────┤ │ │ │ ┌───────────────────────┐ │ │ G1 ──────────────┤ ← G1 既是新生代也是老年代 │ │ │ │ │ (Region 模型, 不分代) │ │ │ │ └───────────────────────┘ │ │ │ │ │ ▼ 老年代收集器 │ │ Serial Old │ │ CMS │ │ Parallel Old │ │ │ │ ZGC / Shenandoah ← 全堆收集, 超低延迟 (JDK 11+ / 15+) │ └─────────────────────────────────────────────────────┘
Serial新生代

Serial 收集器 —— 单线程串行

单线程执行 GC，全程 STW（Stop The World）。简单高效，适合 Client 模式或单核 CPU。

全程 STWJDK 1.3

ParNew 收集器 —— 多线程 Serial

Serial 的多线程版本，GC 时多线程并行。是 JDK 7 之前唯一能和 CMS 搭配的新生代收集器。

STWJDK 1.4
Parallel新生代

Parallel Scavenge 收集器 —— 吞吐量优先

关注可控的吞吐量（用户代码运行时间 / 总时间）。自适应调节策略。JDK 8 默认新生代收集器。

核心参数：-XX:MaxGCPauseMillis 最大停顿时间 | -XX:GCTimeRatio 吞吐量比例

STWJDK 8 默认

Serial Old 收集器

Serial 的老年代版本，单线程 + 标记-整理。一般用于 Client 模式或 CMS 的备用方案。

全程 STW
Parallel Old老年代

Parallel Old 收集器

Parallel Scavenge 的老年代版本，多线程 + 标记-整理。配合 PS 使用，JDK 8 默认组合 Parallel Scavenge + Parallel Old。

STWJDK 8 默认
CMS老年代

CMS 收集器 —— 低延迟优先 ⚠️ JDK 14 已移除

以最短停顿时间为目标，使用标记-清除算法。大部分工作并发执行。

并发收集低延迟JDK 14 移除内存碎片

G1全堆

G1 收集器 —— JDK 9+ 默认，面向大内存

将堆分成多个等大小的 Region，优先回收垃圾最多的 Region（Garbage First）。可设置停顿时间目标。

目标：任意堆大小下停顿 < 1ms，支持 TB 级堆。使用染色指针 + 读屏障。JDK 21 支持分代 ZGC。

5.1 CMS 的 7 个阶段与 3 个致命问题

CMS 工作流程： ① 初始标记 (Initial Mark) → STW, 标记 GC Roots 直接可达, 停顿短 ② 并发标记 (Concurrent Mark) → 并发, 从 GC Roots 遍历对象图, 耗时长 ③ 并发预清理 (Concurrent Preclean) → 并发 ④ 重新标记 (Remark) → STW, 修正并发期间变化, 停顿较长 ⑤ 并发清除 (Concurrent Sweep) → 并发 ⑥ 并发重置 (Concurrent Reset) → 并发 三个致命问题： ❌ 浮动垃圾：并发标记期间新产生的垃圾，本次 GC 无法清除，只能等下次 ❌ 内存碎片：标记-清除产生碎片 → 大对象可能分配失败 → 退化为 Serial Old ❌ Concurrent Mode Failure：并发清理期间老年代满了 → 退化为 Serial Old

5.2 G1 的核心概念：Region

G1 将堆划分为多个大小相等的 Region： ┌───┬───┬───┬───┬───┬───┬───┬───┐ │ E │ E │ S │ E │ O │ H │ E │ E │ ← E=Eden, S=Survivor, O=Old, H=Humongous ├───┼───┼───┼───┼───┼───┼───┼───┤ │ O │ E │ E │ O │ E │ S │ O │ O │ ← 不要求连续，每个 Region 可以随时变换角色 ├───┼───┼───┼───┼───┼───┼───┼───┤ │ E │ O │ E │ O │ O │ E │ E │ O │ └───┴───┴───┴───┴───┴───┴───┴───┘ • Region 大小：1MB ~ 32MB（由 -XX:G1HeapRegionSize 指定，约 2048 个 Region） • Humongous Region：超过 Region 大小一半的对象，连续分配多个 Region • 回收时优先选 垃圾最多的 Region → Garbage First

5.3 GC 分类：Minor GC / Major GC / Full GC / Mixed GC

类型回收区域触发条件STW？
Minor GC (Young GC)仅新生代Eden 满了是（但很短）
Major GC (Old GC)仅老年代老年代空间不足是（CMS 部分并发）
Full GC整个堆 + 元空间老年代满 / 分配担保失败 / Metaspace 不足 / System.gc()是（长！！噩梦！）
Mixed GC (G1 独有)新生代 + 部分老年代 RegionG1 的标记周期完成后是（可控）
Full GC 是性能杀手！一次 Full GC 可能停顿数秒甚至数十秒。JVM 调优的核心目标之一就是避免 Full GC。

-XX:InitiatingHeapOccupancyPercent=45  # 堆占用 45% 触发并发标记周期
-XX:G1ReservePercent=10      # 预留 10% 空间防晋升失败

# ===== 其他重要参数 =====
-XX:MaxTenuringThreshold=15  # 晋升老年代的年龄阈值
-XX:PretenureSizeThreshold=1m # 超过此大小直接进老年代
-XX:+DisableExplicitGC        # 忽略 System.gc() 调用（防止一些库乱触发 Full GC）

八、内存泄漏排查实战

8.1 内存泄漏 vs 内存溢出

内存泄漏 (Memory Leak)

不再使用的对象没有被 GC 回收，占着内存不放。通常由编程错误引起（静态集合、未关闭资源、ThreadLocal 未 remove 等）。

内存溢出 (OOM)

内存真的不够用了。可能是泄漏导致的，也可能是正常业务需要更多内存。泄漏必然导致 OOM，但 OOM 不一定是泄漏。

8.2 常见内存泄漏场景

场景原因解决
静态集合持续添加static Map cache 只增不减用 WeakHashMap / 设置容量上限 / 定期清理
ThreadLocal 未 remove线程池 + ThreadLocal.set() 后不清理try-finally 中 remove
连接/流未关闭Connection、Stream、Channel 未 closetry-with-resources / finally close
内部类持有外部引用非静态内部类隐式持有外部类引用用 static 内部类 + WeakReference
ArrayList 扩容残留remove() 只移元素，不缩容，大数组占着trimToSize() / 新建 list
监听器未注销注册了 Listener/Observer 但未 remove成对注册/注销
动态代理/CGLIB 类膨胀持续生成新代理类，占 Metaspace缓存代理实例 -XX:MaxMetaspaceSize

8.3 排查步骤：拿到 heap dump 后怎么做？

获取 heap dump：jmap -dump:live,format=b,file=heap.hprof [pid] 或用 JVM 参数自动 dump
用 MAT（Memory Analyzer Tool）打开：Eclipse MAT 是最常用的工具，直接拖入 .hprof 文件
看 Leak Suspects 报告：MAT 自动分析→"疑似泄漏"的对象→点击查看"入引用链"
看 Dominator Tree（支配树）：找出"这家伙及其子对象占了多少内存"→找占比最大的
看 GC Roots 路径：从可疑对象出发→"Path to GC Roots"→排除软/弱/虚引用→找到阻止 GC 的根
快速定位技巧：
1. 看 Histogram（类直方图），按 Retained Heap 排序 → 找实例数量异常多的类
2. 如果 byte[] 或 char[] 排第一 → 可能大字符串/大对象
3. 如果自定义类数量远超预期 → 排查创建该类的代码
4. Object[] 排前 → 可能是某个大集合，进去看 contents

8.4 实战模拟：HashMap 导致的内存泄漏

public class CacheService {
    private static final Map<String, Object> cache = new HashMap<>();

    public static void addToCache(String key, Object value) {
        cache.put(key, value);  // 只增不删，永不过期
    }
}

// MAT 排查时你会发现：
//   HashMap$Node[] → Retained Heap 几百 MB
//   → 展开 → 找到是哪个对象的哪个 Map
//   → 看 GC Roots → static CacheService.cache → 解决方案：加 LRU 淘汰

9.3 jstat —— 实时监控 GC 和内存

# jstat -gc [pid] [interval_ms] [count]  — 最常用的命令
$ jstat -gc 8342 1000 10

 S0C    S1C    S0U    S1U      EC       EU        OC         OU       MC     MU
51200 51200    0   24576  409600  307200  2730688   1534464   102400 98200

# 字段含义：
# S0C/S1C   = Survivor 0/1 容量 (Capacity)
# S0U/S1U   = Survivor 0/1 使用量 (Used)
# EC/EU     = Eden 容量/使用量
# OC/OU     = 老年代 容量/使用量   ← 关注 OU/OC: 涨了就停不下来 = 内存泄漏
# MC/MU     = Metaspace 容量/使用量 ← 关注 MU/MC
# YGC/YGCT  = Young GC 次数/总耗时
# FGC/FGCT  = Full GC 次数/总耗时   ← 这个最好为 0!
jstat 关注重点：
· OU（老年代使用量）持续增长，即使 Full GC 也不下降 → 内存泄漏
· FGC 频繁且 FGCT 很大 → 堆太小 / 有泄漏
· MU 接近 MC 且持续增长 → 类加载过多，排查动态代理/CGLIB
· EU 经常满 100% → 对象创建太快 → 扩容新生代或查代码

十一、面试真题速记

JVM 内存模型？堆和栈的区别？
堆：线程共享，存对象实例和数组，GC 主战场。栈：线程私有，存局部变量表、操作数栈、方法出口，一个方法一个栈帧。JDK 8 用元空间替代永久代存类元数据。
Minor GC / Major GC / Full GC 的区别？
Minor GC 只回收新生代（Eden + Survivor），频繁但快。Major GC 回收老年代。Full GC 回收整个堆+元空间，最慢，应尽量避免。G1 还有 Mixed GC（新生代+部分老年代 Region）。
G1 和 CMS 的区别？
CMS 基于标记-清除，老年代是连续内存，有碎片，JDK 14 已移除。G1 基于 Region，不分连续代，标记-整理+复制，无碎片。G1 可以设置停顿时间目标（MaxGCPauseMillis），优先回收垃圾最多的 Region。
什么时候会触发 Full GC？如何避免？
触发：老年代满、Metaspace 满、分配担保失败、System.gc()、jmap -histo:live。避免：Xms=Xmx 防止堆伸缩、设 MaxMetaspaceSize、DisableExplicitGC、提前调大老年代空间、排查内存泄漏。
对象什么时候进老年代？
①熬过 15 次（默认）Minor GC。②大对象（超过 PretenureSizeThreshold）。③ Survivor 放不下。④动态年龄：同年龄对象之和超过 Survivor 一半，该年龄及以上全晋升。
JDK 8 默认是什么 GC？为什么建议升级到 G1？
JDK 8 默认 Parallel Scavenge + Parallel Old。堆 > 4GB 时 Parallel 的 STW 时间不可接受（停顿随堆大小线性增长）。G1 停顿可控（可设目标），4GB+ 堆推荐。
内存泄漏如何排查？
jstat 看老年代使用量是否持续增长→jmap dump→MAT 分析。Leak Suspects 报告→Dominator Tree 找大对象→Path to GC Roots 看谁抓着不放。常见原因：static 集合、ThreadLocal 未 remove、连接未关闭。
CPU 100% 如何排查？
top -Hp 找线程→转 16 进制→jstack 查线程栈→看业务代码是否死循环。如果是 GC 线程 CPU 高 → 频繁 GC → 查 GC 日志 → 可能堆太小或有内存泄漏。
G1 的 Mixed GC 和 Young GC 有什么区别？
Young GC 只回收 Eden + Survivor Region。Mixed GC 在并发标记结束后触发，回收新生代 + 部分垃圾多的老年代 Region。G1 的 Mixed GC 取代了传统 Full GC 的部分职责。
ZGC 和 G1 怎么选？
堆 4GB～16GB、延迟要求 < 200ms → G1 足够。堆 > 16GB、延迟要求 < 10ms → ZGC。JDK 21 的分代 ZGC 是未来趋势，吞吐量也追上了 G1。
