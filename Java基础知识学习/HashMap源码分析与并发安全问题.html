HashMap 源码分析 & 并发访问下的非安全性说明

基于 美团技术博客《Java 8系列之重新认识HashMap》& 酷壳《疫苗：Java HashMap的死循环》整理

一、HashMap 是什么

HashMap 是 Java 中 使用频率最高 的键值对映射容器。它根据 key 的 hashCode 存储数据，O(1) 时间复杂度定位到值，非线程安全。

JDK 版本演变

版本底层结构特点
JDK 1.7数组 + 链表扩容时头插法，并发下可能死循环
JDK 1.8数组 + 链表 + 红黑树尾插法，链表长度 ≥ 8 转红黑树，并发下仍会丢数据

Map 家族对比

实现类线程安全有序性null 键/值适用场景
HashMap❌无序允许 null 键和值通用场景
Hashtable✅ (synchronized)无序不允许遗留类，不推荐
ConcurrentHashMap✅ (CAS+分段锁)无序不允许高并发场景
LinkedHashMap❌插入/访问序允许 null需要有序遍历
TreeMap❌键的自然序不允许 null 键需要排序

二、底层数据结构

2.1 核心字段

// HashMap 内部类 - 哈希桶的基本节点
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;       // 哈希值，用于定位数组索引
    final K key;
    V value;
    Node<K,V> next;       // 链表指针，指向下一个节点
}

// HashMap 关键属性
Node<K,V>[] table;        // 哈希桶数组，2 的 n 次幂
int size;                 // 实际键值对数量
int threshold;            // 扩容阈值 = capacity × loadFactor
final float loadFactor;   // 负载因子，默认 0.75
int modCount;             // 结构修改次数，用于 fail-fast

2.2 为什么数组长度必须是 2 的 n 次幂

常规 Hash 表把桶大小设为素数（如 Hashtable 初始 = 11）来减少冲突。但 HashMap 使用 2 的 n 次幂 有两个原因：

1. 取模优化：h & (length - 1) 代替 h % length，位运算比取模快得多。
2. 扩容优化：扩容时不需要重新计算 hash，只需判断新增的 1 bit 是 0 还是 1。

三、哈希算法

HashMap 的 hash 算法分 三步：

// 第一步：取 hashCode
int h = key.hashCode();

// 第二步：高位参与运算（扰动函数）
// JDK 1.8: 高 16 位异或低 16 位（一次扰动）
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
// JDK 1.7: 四次扰动（多了三次位运算和异或）

// 第三步：取模运算 - 定位数组索引
int index = h & (length - 1);  // 等价于 %，但 & 更快
为什么需要扰动函数？
当 table 长度较小时（如 length=16，n-1=15 即二进制 0000 1111），hash 值只有 低 4 位 参与索引计算。 高位异或低位之后，让高 16 位也参与影响低位，使分布更均匀，减少碰撞。 
示例：length = 16, n-1 = 15 (0000 1111)

没有扰动：
  hash1 = 0001 0101  →  & 0000 1111 = 0101 = 5
  hash2 = 0011 0101  →  & 0000 1111 = 0101 = 5  ← 碰撞！

有扰动（高16位异或低16位后）：
  高位不同 → 异或后低位也不同 → 分布更散

关键源码（JDK 1.8）

final V putVal(int hash, K key, V value, boolean onlyIfAbsent,
               boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    // ① table 为空，扩容
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;
    // ② 计算索引，位置为空，直接新建
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);
    else {
        Node<K,V> e; K k;
        // ③ 首节点 key 相同
        if (p.hash == hash &&
            ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        // ④ 红黑树处理
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        // ⑤ 链表处理（尾插法）
        else {
            for (int binCount = 0; ; ++binCount) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    if (binCount >= TREEIFY_THRESHOLD - 1)
                        treeifyBin(tab, hash);  // ≥8 转红黑树
                    break;
                }
                // key 已存在，跳出准备覆盖
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        // key 已存在，覆盖 value
        if (e != null) {
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null) e.value = value;
            return oldValue;
        }
    }
    ++modCount;
    // ⑥ 超过阈值，扩容
    if (++size > threshold) resize();
    return null;
}

为什么链表长度 ≥ 8 才转红黑树

原因说明
概率极低hash 冲突符合泊松分布，负载因子 0.75 时，桶中节点数达 8 的概率 < 千万分之一
空间成本TreeNode 内存占用是普通 Node 的 2 倍，过早转换浪费空间
平衡点8 是"够晚才转"和"够早优化"之间的 trade-off

五、扩容机制（核心重点）

5.1 JDK 1.7 扩容 —— 头插法（有并发问题）

void resize(int newCapacity) {
    Entry[] oldTable = table;
    int oldCapacity = oldTable.length;
    if (oldCapacity == MAXIMUM_CAPACITY) {
        threshold = Integer.MAX_VALUE;
        return;
    }
    Entry[] newTable = new Entry[newCapacity];  // 新数组
    transfer(newTable);     // ← 关键！数据迁移
    table = newTable;
    threshold = (int)(newCapacity * loadFactor);
}

void transfer(Entry[] newTable) {
    Entry[] src = table;
    int newCapacity = newTable.length;
    for (int j = 0; j < src.length; j++) {
        Entry<K,V> e = src[j];        // 取旧数组每个桶的头节点
        if (e != null) {
            src[j] = null;            // 释放旧数组引用
            do {
                Entry<K,V> next = e.next;    // ① 保存 next
                int i = indexFor(e.hash, newCapacity); // ② 重新计算索引
                e.next = newTable[i];  // ③ 头插！新元素指向链表头
                newTable[i] = e;       // ④ 新元素成为链表头
                e = next;              // ⑤ 继续处理下一个
            } while (e != null);
        }
    }
}
JDK 1.7 扩容特点：
· 头插法：新元素始终放在链表头部
· 迁移后链表 顺序反转
· 每次 rehash 都需要 重新计算索引
· 这就是导致并发死循环的根本原因

5.2 JDK 1.8 扩容 —— 尾插法 + 无需 rehash

final Node<K,V>[] resize() {
    // ... 计算新容量 newCap = oldCap << 1（翻倍）...

    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    table = newTab;

    if (oldTab != null) {
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;

                // 只有一个节点，直接迁移
                if (e.next == null)
                    newTab[e.hash & (newCap - 1)] = e;

                // 红黑树分裂
                else if (e instanceof TreeNode)
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);

                // 链表：不用 rehash！只判断新增 1 bit
                else {
                    Node<K,V> loHead = null, loTail = null; // 留在原位
                    Node<K,V> hiHead = null, hiTail = null; // 移到 j+oldCap

                    Node<K,V> next;
                    do {
                        next = e.next;
                        // 关键：e.hash & oldCap，只判断新增的那 1 bit
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null) loHead = e;
                            else loTail.next = e;   // 尾插！
                            loTail = e;
                        } else {
                            if (hiTail == null) hiHead = e;
                            else hiTail.next = e;   // 尾插！
                            hiTail = e;
                        }
                    } while ((e = next) != null);

                    if (loTail != null) { loTail.next = null; newTab[j] = loHead; }
                    if (hiTail != null) { hiTail.next = null; newTab[j + oldCap] = hiHead; }
                }
            }
        }
    }
    return newTab;
}

JDK 1.8 扩容为什么不需要 rehash

核心原理：
原容量 16 = 10000₂，扩容后 32 = 100000₂

原索引 = hash & 15（低 4 位）
新增判断 = hash & 16（第 5 位）

· 第 5 位 = 0 → 索引不变
· 第 5 位 = 1 → 索引 = 原索引 + 16

示例：hash = 21 (10101₂)
原索引 = 21 & 15 = 10101 & 01111 = 5
新位 = 21 & 16 = 10101 & 10000 = 16 ≠ 0
新索引 = 5 + 16 = 21

5.3 JDK 1.7 vs JDK 1.8 扩容对比

对比维度JDK 1.7JDK 1.8
插入方式头插法尾插法
迁移后链表顺序反转保持原序
rehash 计算每个元素重新计算只判断新增 1 bit
并发扩容可能死循环！不会死循环，但会丢数据
红黑树无≥8 转树，≤6 退化回链表

六、并发访问下的非安全性（重点）

6.1 JDK 1.7 死循环问题 —— CoolShell 经典分析

触发条件：多线程同时 put 触发 resize，两个线程同时执行 JDK 1.7 的 transfer() 方法（头插法）。

复现代码

public class HashMapInfiniteLoop {
    // 初始长度 2，threshold = 2 × 0.75 = 1
    // put 第二个元素就触发 resize
    private static HashMap<Integer,String> map = new HashMap<>(2, 0.75f);

    public static void main(String[] args) {
        map.put(5, "C");

        new Thread("T1") {
            public void run() { map.put(7, "B"); }
        }.start();

        new Thread("T2") {
            public void run() { map.put(3, "A"); }
        }.start();
    }
}

死循环推演（逐步跟进，理解每一步）

阶段 0：初始状态
旧表 (size=2):
table[0] → null
table[1] → (5,C) → null                  假设 hash(5)%2=1

此时 put(7,B)，hash(7)%2=1，仍在 table[1]：
table[1] → (7,B) → (5,C) → null          头插法，7 是新元素

put(3,A)，hash(3)%2=1，还在 table[1]！
但 table[1] 已经有 2 个元素，超过 threshold=1 → 扩容到 4
阶段 1：线程 2 先完成扩容（正常流程）
线程 2 把旧表迁移到新表（长度 4），rehash 后：
（假设 hash(3)%4=3, hash(7)%4=3, hash(5)%4=1）

新表 (线程2, size=4):
table[1] → (5,C) → null
table[3] → (3,A) → (7,B) → null
           ↑ 注意！头插法 → 3先插入，7插到头 → 3成了7的next
阶段 2：线程 1 被调度回来 —— 关键！
线程 1 在 transfer() 中被挂起时的本地变量：
  e    = key(3)     ← 指向旧表中的 key(3)
  next = key(7)     ← 在 transfer 第一步 e.next 保存的

但！在线程 2 已经完成的新表里：
  key(7).next = key(3)    ← 链表顺序已被线程2反转！
这就是灾难的种子：线程 1 的 next 指向 key(7)，而 key(7).next 却被线程 2 改成了 key(3)。
线程 1 对此一无所知，它将继续用头插法迁移这条已经被反转的链表。
阶段 3：线程 1 继续执行 —— 环形链表形成
循环第 1 次（e = key(3)）：
  ① next = e.next → next = key(7)
  ② 计算索引 i = indexFor(3) = 3
  ③ e.next = newTable[3]  → key(3).next = null     (newTable[3] 初始为空)
  ④ newTable[3] = e       → newTable[3] = key(3)
  ⑤ e = next              → e = key(7)

  newTable[3] → key(3) → null

循环第 2 次（e = key(7)）：
  ① next = e.next → next = key(3)  ← 线程2让key(7).next指向key(3)！！
  ② 计算索引 i = indexFor(7) = 3
  ③ e.next = newTable[3]  → key(7).next = key(3)   (key(3)是当前链表头)
  ④ newTable[3] = e       → newTable[3] = key(7)
  ⑤ e = next              → e = key(3)

  newTable[3] → key(7) → key(3) → ???

循环第 3 次（e = key(3)）：
  ① next = e.next → next = null     (key(3).next在第1次循环被设为null)
  ② 计算索引 i = indexFor(3) = 3
  ③ e.next = newTable[3]  → key(3).next = key(7)   ← ← ← 环形！！！
  ④ newTable[3] = e       → newTable[3] = key(3)
  ⑤ e = next              → e = null（循环结束）

  最终：newTable[3] → key(3) → key(7) → key(3) → key(7) → ... ♻
♻ 环形链表形成
   key(3) ──→ key(7)
      ↑          │
      └──────────┘

   任何 get() 操作落到 table[3]
   → 遍历链表永远找不到 null
   → Infinite Loop → CPU 100%

6.2 JDK 1.8 的改进

JDK 1.8 把 头插法改为尾插法，扩容迁移时保持链表原有顺序，因此 不会出现死循环。

// JDK 1.8 尾插法（关键区别！）
if (loTail == null)
    loHead = e;        // 第一个节点 = 链表头
else
    loTail.next = e;   // 追加到链表尾部 ← 不是头插！
loTail = e;            // 更新尾指针

6.3 但 JDK 1.8 仍然有并发安全问题！

问题 1：数据丢失
两个线程同时 put 同一个桶的尾部（链表未转树），后执行的线程会覆盖先执行线程的新节点。 
// 线程 A 和 B 同时 put 到同一个桶的链表尾部
线程 A: p.next = newNode(...)   // 在链表尾部添加节点 A
线程 B: p.next = newNode(...)   // 也在链表尾部添加节点 B
// 后执行的覆盖先执行的 → 一个节点丢失
问题 2：size 计数不准确
size++ 不是原子操作，两个线程同时 put 后 size 可能只 +1（实际应该 +2），导致扩容时机延后。 
线程 A: size = 5 → 6
线程 B: size = 5 → 6   ← 两个 put，size 只加了 1
实际元素数 7 > size 6 → 扩容延后 → 链表变长 → 性能下降
问题 3：get 获取不到刚 put 的数据
线程 A 刚 put(key, value)，但还未刷新到主内存；线程 B 读的是旧的 table 状态，返回 null。

6.4 解决方案

方案适用场景推荐度
ConcurrentHashMap高并发读写✅ 首选
Collections.synchronizedMap(new HashMap<>())低并发⚠️ 可用，性能差
Hashtable遗留兼容❌ 不推荐
ConcurrentHashMap 核心设计（JDK 1.8）：
· CAS + synchronized 只锁链表头节点（锁粒度 = 一个桶，不是整个表）
· 读操作完全无锁（volatile 保证可见性）
· 扩容时多线程协同迁移（ForwardingNode 标记已迁移的桶）

七、核心面试速记

HashMap 底层结构？
JDK 1.7：数组+链表；JDK 1.8：数组+链表+红黑树（≥8 转树，≤6 退化）
为什么用红黑树不用 AVL 树？
红黑树插入删除旋转次数少（最多 3 次），AVL 最坏需 O(logn) 次旋转
为什么数组长度必须是 2 的 n 次幂？
h & (n-1) 代替取模加速；扩容时不需要 rehash，只判断新增 1 bit
为什么负载因子默认 0.75？
时间与空间折中；泊松分布下桶中节点 ≥ 8 概率 < 千万分之一
扩容为什么是 2 倍？
与 "2^n 长度" 配合：扩容后索引 = 原位置 or 原位置+oldCap，无需重新 hash
put 方法的完整流程？
hash → 定位索引 → 空则新建 → 碰撞则查链表/红黑树 → 覆盖或新增 → 判断扩容
JDK 7 并发死循环根因？
头插法 + 扩容迁移 → 两个线程同时 transfer → 链表指针混乱 → 环形链表
JDK 8 还有并发问题吗？
不会死循环（改尾插法），但仍有 数据覆盖、size 不准、get 为 null 等问题
线程安全应该选什么？
ConcurrentHashMap（CAS + 分段锁，读无锁，写只锁一个桶）
